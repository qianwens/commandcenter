Data Storage & External Services

The application persists customer, account, account type, and transaction records in a SQL Server database (ZavaBankDB). It also calls an external ledger API for account balances and transactions, and redirects users to an external authentication gateway for sign-in/sign-out flows.

Key Architectural Decisions

  • Uses page-level code-behind rather than layered controller/service abstractions.
  • Uses direct SQL statements through ADO.NET instead of an ORM.
  • Implements a hybrid transaction-history flow: external ledger first, SQL fallback on failure.

Core Business Workflows

The application supports teller/admin account-management workflows: selecting customers, opening/closing accounts, and viewing balances and transactions with graceful fallback behavior when external services are unavailable.

Primary Workflows

Workflow 1: Browse Customer Accounts

Staff opens Default.aspx, the system checks authentication, loads a customer list, then loads selected customer accounts. This workflow applies access-control rules first and then uses SQL reads to present current operational data.

Workflow 2: Open New Account

Staff submits the open-account form. The application validates customer id, account type, and opening balance, creates a generated account number, inserts a new row in Accounts, and refreshes the customer account list.

Workflow 3: Review Balance and Transaction History

Staff requests balance or transactions for an account. Balance is fetched from the ledger service. Transaction history is fetched from ledger first; if that call fails, local SQL transactions are shown as a continuity fallback.

Cross-Service Data Flows

The UI service composes account views from local SQL ownership data plus externally sourced ledger data. Data composition occurs per selected account: account metadata is sourced locally, while latest balance and recent transactions are requested from the ledger API and merged into the page response. If the ledger transaction call fails, business output degrades gracefully by returning locally stored transaction history.

Business Rules & Decision Logic

  • Access rule: unauthenticated users are redirected to login with return URL preservation.
  • Input rule: open-account action requires valid customer ID, account type ID, and decimal opening balance; invalid input cancels insertion.
  • Account lifecycle rule: close-account command transitions account status to Closed and stamps close/modified dates.
  • Data continuity rule: transaction-history retrieval prefers ledger data but falls back to local SQL when remote retrieval fails.

No explicit cache provider or framework-level cache annotations were detected. The only optimization is conditional fallback from remote ledger API calls to local SQL transaction reads, which improves workflow continuity but is not cache-aside/read-through caching.

Data Ownership Boundaries
