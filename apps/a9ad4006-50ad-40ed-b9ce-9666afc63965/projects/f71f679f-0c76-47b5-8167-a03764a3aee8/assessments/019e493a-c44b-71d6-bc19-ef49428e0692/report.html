Data Storage & External Services

The application does not maintain any local persistent storage. All state lives in external systems: RabbitMQ acts as the durable message broker holding the q.notify.email queue (with a configured dead-letter exchange zava.dlx for failed messages), and the SMTP server (defaulting to MailHog for development) delivers outbound email. Configuration is read from app.config with environment-variable overrides to support containerised deployments.

Key Architectural Decisions

  • Polling over push: The worker uses a synchronous BasicGet poll loop with a configurable Worker.PollIntervalMs interval rather than a push-based IBasicConsumer, keeping the implementation simple and avoiding threading complexity.
  • Dual-format message parsing: Messages can arrive as either JSON or XML; the parser auto-detects the format by inspecting the first character of the payload, allowing integration with heterogeneous upstream producers.
  • Dead-letter handling via nack: Processing failures result in a BasicNack with requeue=false, routing the message to the configured dead-letter exchange so failed notifications are not silently dropped.

Communication Patterns

Asynchronous inbound (AMQP / RabbitMQ):

The worker uses a synchronous poll (BasicGet) rather than a push-based consumer. Each poll cycle opens a new TCP connection and channel to RabbitMQ, retrieves at most one message, processes it, then closes the connection. The poll interval is configurable (Worker.PollIntervalMs, default 5 000 ms). On successful processing the message is acknowledged (BasicAck); on failure it is negatively acknowledged with requeue=false (BasicNack), routing it to the configured dead-letter exchange (zava.dlx).

Synchronous outbound (SMTP):

Email delivery uses System.Net.Mail.SmtpClient in a blocking, synchronous call. There is no retry policy, timeout configuration, or circuit breaker — a transient SMTP failure will cause a single exception and the message will be nack'd to the dead-letter exchange.

Resilience:

No circuit breaker, retry, or timeout policy is implemented. The top-level poll loop catches all exceptions, logs them, and continues polling after the configured interval, providing a basic crash-recovery loop at the process level only.

Service discovery:

All endpoints (RabbitMQ host/port and SMTP host/port) are resolved from static configuration (environment variables with app.config fallback). No dynamic service discovery (Consul, Eureka, Kubernetes DNS) is used.

Security posture:

  • RabbitMQ connection uses username/password credentials (zava_app / zava_pass by default). No TLS is configured on the AMQP connection (AutomaticRecoveryEnabled = false; no SslOptions set).
  • SMTP uses UseDefaultCredentials = true with no explicit credentials or TLS/STARTTLS configuration.
  • No authentication or transport security (TLS) is configured for either outbound channel; credentials are passed in plaintext. This is a security concern for any non-loopback network deployment.

Core Business Workflows

ZavaNotifyWorker fulfils the notification delivery responsibility for the ZavaBank platform: it receives transaction alert requests from upstream banking services via a message queue and delivers personalised email notifications to customers.

This is a single-domain, single-service application. The only domain object is the NotificationMessage, which is not persisted and has no relationships with other stored entities.

ServiceDomain ContextOwned EntitiesExternal Dependencies

Primary Workflows

Workflow 1: Notification Poll and Email Dispatch

This is the sole business workflow. The worker runs a continuous poll loop that processes one notification message per cycle.

Trigger: Timer-driven — the poll loop wakes every Worker.PollIntervalMs milliseconds (default 5 000 ms).

Steps:

  1. Poll queue — The worker calls BasicGet on queue q.notify.email. If no message is available, the cycle ends and the worker sleeps until the next interval.
  2. Parse payload — The message body is examined:
    • If the body starts with <, it is treated as XML and parsed using XDocument with support for element name aliases (e.g., AlertType / Type, TransactionId / TransactionID).
    • Otherwise it is treated as JSON and parsed using JavaScriptSerializer with support for key aliases (e.g., alertType / AlertType / type).
  1. Validate recipient — If CustomerEmail is absent or whitespace, an InvalidOperationException is thrown immediately — the message cannot be delivered.
  2. Compose email — Subject and plain-text body are formatted using the fields from NotificationMessage (alert type, customer ID/name, transaction ID, amount, currency, details, timestamp).
  3. Dispatch email — SmtpClient.Send delivers the email to the configured SMTP server.
  4. Acknowledge or dead-letter — On success: BasicAck removes the message from the queue. On any exception during steps 2–5: BasicNack with requeue=false routes the message to the dead-letter exchange zava.dlx.

Cross-Service Data Flows

ZavaNotifyWorker is a terminal consumer — it does not call any other service to enrich or aggregate data. All information required to send the email must be present in the message payload itself.

Upstream banking services (transaction processor, fraud detection, etc.) are responsible for constructing complete NotificationMessage payloads before publishing to q.notify.email. The contract between producers and this worker is implicit (no schema registry or OpenAPI contract) and supports two formats (JSON and XML) with flexible field name aliases.

When delivery fails (SMTP unavailable, malformed payload, missing email address), the message is routed to zava.dlx. Recovery from the dead-letter exchange is not handled by this service — it requires manual intervention or a separate dead-letter consumer (not present in this repository).

Business Rules & Decision Logic

Validation Rules:

  • Recipient required: CustomerEmail must be a non-null, non-whitespace string. If absent, processing is aborted and the message is dead-lettered. No other fields are validated for presence — missing values default to "N/A" in the email body.
  • Payload format detection: The first non-whitespace character of the message body determines the parser. A < character triggers XML parsing; anything else triggers JSON parsing. There is no explicit schema validation — malformed payloads will raise a parse exception and be dead-lettered.

Decision Logic:

  • Alert type fallback: If AlertType is empty, the email subject reads "Zava Bank Notification: Transaction ..." rather than using the alert type.
  • Transaction ID fallback: If TransactionId is empty, the subject displays "N/A" in place of the transaction reference.
  • Field alias resolution: Both JSON and XML parsers try multiple field name variants (e.g., customerId / CustomerId / userId) in priority order, taking the first non-null match. This allows the worker to accept messages from producers using different naming conventions.

Error Handling:

  • All exceptions within a poll cycle are caught at two levels:
  1. Processing level (PollQueue): parse or send failures cause BasicNack(requeue=false), routing to the dead-letter exchange.
  2. Loop level (Main): any uncaught exception from PollQueue (e.g., RabbitMQ connection failure) is logged to stdout and the loop continues after the poll interval.
  • There is no retry policy, back-off strategy, or maximum dead-letter requeue limit defined within this service.

Audit / Logging:

  • Successful deliveries are logged to stdout with CustomerEmail, CustomerName, CustomerId, AlertType, TransactionId, Amount, and Currency — note this logs PII and financial data in plaintext.
  • Failures are logged with the exception message only (no stack trace).
  • No structured logging, correlation IDs, or audit trail is maintained.

Authorization:

  • No authentication or authorisation is applied. Any process able to publish to q.notify.email can trigger email delivery to arbitrary addresses. There is no sender allowlist, rate limiting, or abuse prevention mechanism.

Version & Compatibility Risks

The project targets .NET Framework 4.8, which is in long-term servicing mode on Windows but is not forward-compatible with .NET 6/7/8. RabbitMQ.Client 5.2.0 was released in 2019 and is several major versions behind the current release (7.x as of 2025); the 5.x line is no longer maintained and lacks support for modern AMQP features, async consumers, and important security patches. The application runs on Mono 6.12 inside Docker (see Dockerfile), which is an unmaintained runtime — Mono reached end-of-life and Microsoft recommends migrating to .NET 6+. System.Web.Script.Serialization (JavaScriptSerializer) is a legacy serialiser present only in .NET Framework and has been superseded by System.Text.Json or Newtonsoft.Json.

Notable Observations

  • Mono container runtime is end-of-life: The Dockerfile uses mono:6.12 as the base image. Mono is no longer maintained by Microsoft; the recommended migration path is to .NET 8 or later running on the official mcr.microsoft.com/dotnet/runtime image.
  • RabbitMQ.Client 5.2.0 is significantly out of date: The current stable line is 7.x, which introduces async-first APIs, improved connection recovery, and OAuth2 support. The 5.x assembly reference points to the net451 target, which may cause compatibility issues on modern runtimes.
  • JavaScriptSerializer is framework-locked: System.Web.Script.Serialization is not available outside .NET Framework. Migrating to .NET requires replacing it with System.Text.Json or Newtonsoft.Json.
  • No logging framework: Observability relies entirely on Console.WriteLine. A structured logging library (e.g., Serilog, NLog, Microsoft.Extensions.Logging) is absent, limiting production diagnostics.

Caching Strategy

No caching layer is implemented. The application is stateless between poll cycles; there are no in-process caches, distributed caches, or query result caches. Each poll cycle independently opens a fresh connection to RabbitMQ and retrieves a single message.

Data Ownership Boundaries

The application owns no data stores. Data boundaries are:

  • SMTP server (external, infrastructure-owned): Email delivery and storage of sent messages are entirely external.

There is no cross-service data access pattern within this repository — the worker is a terminal consumer that reads from a queue and writes to an email relay.
