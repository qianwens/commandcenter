The application persists data in SQL Server and reads DB connectivity from either environment variables or the web.config connection string. No cache, queue, or outbound third-party API integration is present in the codebase.

Key Architectural Decisions

  • Uses a single service-class design (StatementService) with direct SQL access instead of a repository abstraction.
  • Uses WCF basicHttpBinding for compatibility-oriented SOAP exposure.
  • Adds a lightweight unauthenticated health endpoint for runtime liveness checks.

Core Business Workflows

The application supports monthly bank statement generation and statement retrieval for account holders. Its core workflow converts account and transaction data into persisted HTML statements and archive metadata.

Primary Workflows

Workflow 1: Generate Monthly Statement

Entry point: SOAP operation GenerateStatement(accountId, month, year).

  1. Validate account ID and requested period boundaries.
  2. Load account and customer identity data.
  3. Load all in-period transactions and compute credit/debit totals.
  4. Build formatted HTML statement output.
  5. Persist statement request, generated statement content, and archive metadata/checksum.
  6. Return generated statement payload to caller.

Business rules involved: account and date validation, transaction credit/debit classification, deterministic HTML and checksum generation.

Workflow 2: Retrieve Statement History and Detail

Entry points: SOAP operations GetStatementHistory(accountId) and GetStatement(statementId).

  1. Validate identifiers.
  2. Query persisted generated statements.
  3. Return history list (top 100 recent records) or full statement detail.
  4. Return not-found faults when data is unavailable.

Cross-Service Data Flows

No multi-service composition pattern exists in this repository. The service pulls and writes all required data directly from/to SQL Server within one process boundary. If the database is unavailable, workflows fail immediately; no fallback data source or degraded partial response path is implemented.

Business Workflow Sequence

sequenceDiagram participant Caller participant Service as "StatementService" participant DB as "SQL Server" Caller->>Service: Request monthly statement Service->>Service: Validate account and period Service->>DB: Load account and customer DB-->>Service: Account context Service->>DB: Load period transactions DB-->>Service: Transaction list Service->>Service: Compute totals and render statement Service->>DB: Store request and generated statement Service->>DB: Store archive checksum metadata DB-->>Service: Persisted identifiers Service-->>Caller: Generated statement response

Business Rules & Decision Logic

  • Input rules: accountId and statementId must be positive, month must be 1-12, year must be 2000-2100.
  • Credit/debit rule: transactions are treated as credits when type code is DEP or amount is non-negative; otherwise debits are absolute-value aggregated.
  • State mutation: generating a statement creates new records across request, generated statement, and archive tables in sequence.
  • Data integrity: service ensures GeneratedStatements table exists before operations; not-found conditions return service faults.
  • Authorization and role-based business policies are not implemented in service code or configuration.
