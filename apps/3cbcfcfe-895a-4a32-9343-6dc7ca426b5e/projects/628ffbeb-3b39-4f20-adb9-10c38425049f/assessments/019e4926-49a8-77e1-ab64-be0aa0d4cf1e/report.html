Core Business Workflows

The service supports bank-statement generation and retrieval workflows for account holders by assembling transaction data and persisting generated statement artifacts.

Primary Workflows

Workflow 1: Generate Monthly Statement

The workflow starts when a caller submits GenerateStatement(accountId, month, year). The service validates period boundaries, loads account and transaction data, computes credits/debits, renders statement HTML, and persists request, generated, and archive records before returning the generated statement response.

Workflow 2: View Statement History

The caller invokes GetStatementHistory(accountId). After validating the account identifier, the service returns the most recent generated statements for that account from GeneratedStatements.

Workflow 3: Retrieve Statement Detail

The caller invokes GetStatement(statementId). The service validates the input, loads one statement row, and returns statement content and metadata.

Cross-Service Data Flows

No cross-service aggregation flow is implemented. All workflows execute inside one service and one SQL Server data source. If the database is unavailable, statement operations fail with service faults and no fallback path is configured.

Business Workflow Sequence

sequenceDiagram participant User participant Service as "StatementService" participant DB as "SQL Server" User->>Service: Request monthly statement Service->>Service: Validate account and period rules Service->>DB: Fetch account and transaction data DB-->>Service: Account snapshot and transactions Service->>Service: Compute totals and build statement content Service->>DB: Persist request, generated statement, and archive DB-->>Service: Stored identifiers Service-->>User: Generated statement response

Business Rules & Decision Logic

  • Input validation rules: accountId must be positive, month must be 1-12, and year must be in accepted range.
  • Statement totals are derived from transaction type and signed amount values.
  • Statement generation guarantees persistence of request and archive metadata before response completion.
  • Retrieval workflows enforce existence checks and return faults when records are missing.
