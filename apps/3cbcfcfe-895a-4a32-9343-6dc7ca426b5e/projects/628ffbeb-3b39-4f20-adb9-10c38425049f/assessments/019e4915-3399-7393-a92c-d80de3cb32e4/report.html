Data Storage & External Services

The application has a single data dependency: a SQL Server instance (ZavaBankDB) accessed via ADO.NET (SqlConnection / SqlCommand). Connection details are resolved first from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) and fall back to the ZavaBankDb connection string in web.config. There are no caches, message brokers, or other external API integrations.

Key Architectural Decisions

  • WCF over basicHttpBinding — service operations are exposed as a SOAP endpoint, making the service callable by any WS-* compatible client without a proprietary SDK.
  • Raw ADO.NET without an ORM — SqlConnection and SqlCommand are used directly; there is no Entity Framework or Dapper layer, giving full control over SQL but requiring manual schema management (e.g., EnsureGeneratedStatementsTableExists).
  • Environment-variable-first configuration — DbConfig prefers environment variables over web.config connection strings, enabling twelve-factor-style deployment in containers.

Communication Patterns

Synchronous communication only. All three service operations are synchronous request/reply over basicHttpBinding (SOAP 1.1 over HTTP). There is no asynchronous messaging, event-driven pattern, message queue, or pub/sub integration.

Data access is direct ADO.NET over TCP to SQL Server — synchronous SqlCommand.ExecuteReader() / ExecuteScalar() calls with no connection pooling configuration beyond ADO.NET defaults.

Resilience patterns: None. There is no retry policy, circuit breaker, timeout configuration, or bulkhead. A SQL Server connection failure will propagate directly to the SOAP client as a FaultException.

Service discovery: None. The SQL Server address is resolved from environment variables (DB_HOST, DB_PORT) or a hardcoded web.config connection string (Server=sqlserver,1433). There is no Consul, Eureka, or Kubernetes DNS-based service discovery.

API gateway: None. The WCF service is directly client-facing with no gateway, load balancer, or reverse proxy defined in the repository.

Security posture: No authentication or TLS is configured. The WCF endpoint uses basicHttpBinding with no <security> element, meaning transport security (HTTPS) and message-level security are both disabled. All three operations and the health endpoint are publicly accessible with no authentication or authorization checks. The web.config allows anonymous access to the /health path explicitly (<allow users="*" />).

Primary Workflows

Workflow 1: Generate Monthly Statement

Trigger: SOAP client calls GenerateStatement(accountId, month, year).

Steps:

  1. Validate request inputs — enforce that accountId > 0, month is 1–12, and year is in range 2000–2100.
  2. Compute billing period — derive periodStart (first day of month) and periodEnd (last instant of month).
  3. Look up account and customer — query Accounts joined with Customers to obtain AccountNumber, current Balance, and full customer name. Raise a fault if the account is not found.
  4. Fetch transactions for the period — query Transactions joined with TransactionTypes for all transactions within [periodStart, periodEnd], ordered chronologically. Accumulate totalCredits and totalDebits.
  5. Classify credits vs debits — transactions where TypeCode = "DEP" or Amount >= 0 are credits; all others contribute to debits.
  6. Render HTML statement — compose a complete HTML document embedding customer details, account number, period dates, opening balance, totals, and a transaction table.
  7. Compute checksum — generate a SHA-256 hash of the HTML content for tamper-detection.
  8. Record the request — insert a row into StatementRequests (status Completed, format HTML).
  9. Persist the statement — insert the HTML into GeneratedStatements, capturing the new StatementID.
  10. Archive metadata — insert into StatementArchive with a 7-year expiry date and the SHA-256 checksum.
  11. Return response — return GeneratedStatementResponse including the StatementId and the full HTML content.

Workflow 2: Retrieve Statement History

Trigger: SOAP client calls GetStatementHistory(accountId).

Steps:

  1. Validate input — enforce that accountId > 0.
  2. Ensure statement table exists — auto-create GeneratedStatements table if absent.
  3. Query recent statements — fetch up to 100 most recently generated statements for the account, ordered by GeneratedDate DESC.
  4. Return summary list — return StatementHistoryResponse with lightweight StatementHistoryItem records (ID, month, year, generated date) — no HTML content.

Workflow 3: Retrieve a Specific Statement

Trigger: SOAP client calls GetStatement(statementId).

Steps:

  1. Validate input — enforce that statementId > 0.
  2. Ensure statement table exists — auto-create GeneratedStatements table if absent.
  3. Look up statement — query GeneratedStatements by StatementID.
  4. Return full detail or fault — if found, return StatementDetailResponse with full HTML content; if not found, raise FaultException("Statement was not found.").

Cross-Service Data Flows

ZavaStatementService is a single-service application — there are no remote service calls, gateway aggregation patterns, or event-driven cross-service compositions. All data originates from a single shared SQL Server database.

The only cross-boundary flow is the implicit dependency on the upstream banking schema:

  • GenerateStatement reads from the upstream banking tables (Accounts, Customers, Transactions, TransactionTypes) to gather the source data for the statement.
  • It then writes to the statement-layer tables (StatementRequests, GeneratedStatements, StatementArchive) which this service owns.
  • There is no circuit-breaker or fallback: if the upstream tables are unavailable or the account does not exist, the operation fails immediately with a FaultException.

Credit/Debit Classification

A transaction is classified as a credit if its TypeCode equals "DEP" (case-insensitive) or its Amount is greater than or equal to zero. Otherwise it is classified as a debit and its absolute value is added to totalDebits. This dual-condition rule means that non-deposit positive-amount transactions are also treated as credits.

Archive Retention Policy

Every generated statement is archived with an expiry of exactly 7 years (DATEADD(YEAR, 7, GETDATE())). The archive record stores a SHA-256 checksum of the HTML content for tamper-detection and a FilePath reference (db://generated-statements/<statementId>).

Statement History Cap

GetStatementHistory returns at most 100 statements per account (SELECT TOP 100 ... ORDER BY GeneratedDate DESC), providing the most recent statements only.

Schema Auto-Initialization

Before any read or write against GeneratedStatements, the service executes a DDL guard (IF OBJECT_ID ... IS NULL CREATE TABLE) to ensure the table exists. This means the schema for the statement layer is lazily initialized on first use rather than through a migration tool.

Cross-Cutting Concerns

  • Transactions: There is no explicit transaction scope (TransactionScope, BEGIN TRANSACTION). The three INSERT statements in GenerateStatement (StatementRequest → GeneratedStatements → StatementArchive) execute in sequence without wrapping. A failure on the second or third INSERT leaves partial data committed.
  • Error handling: All business errors propagate as FaultException (untyped), returning a SOAP Fault to the client. There are no typed fault contracts (FaultException<T>), no retry logic, and no compensating actions.
  • Audit/logging: No explicit logging framework is used. Statement requests are tracked implicitly in StatementRequests with RequestedDate and CompletedDate timestamps. No structured audit trail or event log is written.
  • Authorization: No authentication or authorization is enforced. Any caller with network access can generate or retrieve statements for any account ID. There are no role checks, ownership validation, or access control lists.

Version & Compatibility Risks

All referenced assemblies ship as part of .NET Framework 4.8, which Microsoft has placed in maintenance mode (security fixes only, no new features). .NET Framework 4.8 will not be ported to non-Windows platforms natively — the project currently works on Linux only via Mono 6.12, which is also in maintenance mode and reached end-of-active-development. System.ServiceModel (WCF) is the most significant migration risk: Microsoft does not ship a WCF server runtime for modern .NET (5/6/7/8); migrating would require replacing WCF with ASP.NET Core gRPC, REST APIs, or the community-maintained CoreWCF library. System.Data.SqlClient has been superseded by Microsoft.Data.SqlClient, which receives active development, security patches, and new SQL Server feature support.

Notable Observations

  • Zero third-party NuGet packages — the project relies entirely on .NET Framework BCL assemblies, which significantly reduces supply-chain risk but also means the application is tightly coupled to the full .NET Framework runtime.
  • Mono runtime dependency — the Dockerfile uses mono:6.12 + xsp4 to run a .NET Framework app on Linux; Mono is no longer actively developed and is not a supported production platform for new workloads.
  • No logging framework — there is no structured logging dependency (e.g., Serilog, NLog, log4net); all diagnostic output is implicit through WCF fault exceptions and IIS/XSP4 request logs.
  • System.Data.SqlClient (legacy) — the project uses the older System.Data.SqlClient namespace rather than the actively maintained Microsoft.Data.SqlClient NuGet package, missing security patches, AAD auth support, and Always Encrypted improvements.

Caching Strategy

No caching layer is configured. There is no Redis, EhCache, Caffeine, MemoryCache, IDistributedCache, or any query-result caching. Every call to GenerateStatement, GetStatementHistory, and GetStatement executes fresh SQL queries against SQL Server. Repeatedly retrieving the same statement will re-query GeneratedStatements each time.

Data Ownership Boundaries

Single shared database. All seven tables reside in the same SQL Server database (ZavaBankDB). There is no logical or physical schema separation between the upstream banking tables (Accounts, Customers, Transactions, TransactionTypes) and the statement-layer tables (StatementRequests, GeneratedStatements, StatementArchive).

Read-only vs read-write split. ZavaStatementService treats the four banking tables as read-only (SELECT only) and owns the three statement tables exclusively (INSERT and DDL). There is no formal ownership enforcement (foreign key constraints to banking tables are not defined in the code).

No CQRS or event sourcing. All read and write operations go through the same synchronous ADO.NET path to the same SQL Server instance. There is no separate read model, event store, or command/query separation.
