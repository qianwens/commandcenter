Data Storage & External Services

The application persists operational data in a single SQL Server database (ZavaBankDB) that stores fraud alerts, rules, transactions, and user/session data. It also depends on an external authentication gateway endpoint (WhoAmI.ashx) to translate .NET authentication cookies into a session token.

Key Architectural Decisions

  • Uses Struts interceptor-based session enforcement (AuthTokenInterceptor) to guard all actions except login and health.
  • Keeps all persistence access in action classes through direct JDBC calls rather than a dedicated repository/service layer.
  • Integrates with an external SSO gateway to bridge Java and .NET authentication cookies.

DTOs & Contracts

FraudAlertRecord and FraudRuleRecord are mutable response models used between action classes and JSP views. SessionUser is an immutable authentication session contract passed through Struts session state. No OpenAPI, protobuf, or GraphQL contract files were found. Serialization for internal auth integration is manual string parsing of the sessionToken value returned by WhoAmI.ashx.

Communication Patterns

The system is primarily synchronous: browser requests hit Struts actions, actions execute SQL through JDBC, and results are rendered in JSP pages. SsoSessionService performs an additional synchronous HTTP call to the auth gateway when it must resolve a .NET auth cookie. No asynchronous messaging, circuit breaker library, or retry policy framework is configured; timeout handling for the auth gateway call is fixed to 3 seconds. Service discovery is not used, and downstream URLs are configured directly. TLS/authz enforcement at API contract level is not explicitly configured in this codebase; session validation is handled by custom interceptor logic.

Core Business Workflows

ZavaFraudDetector supports fraud operations staff in reviewing flagged transactions and maintaining fraud detection rules. Workflows center on session-authenticated analyst actions and SQL-backed decision updates.

Primary Workflows

Workflow 1: Analyst authentication and queue access

  1. Analyst opens login or queue page.
  2. Interceptor/service attempts to resolve analyst session from session state, request token, header token, or auth cookie.
  3. If only auth cookie is present, app calls external auth gateway to resolve session token.
  4. Session token is validated against active token and user records in SQL Server.
  5. On success, user context is stored in session and analyst is redirected to the flagged queue.

Workflow 2: Fraud alert decision processing

  1. Analyst opens flagged queue and selects an alert.
  2. Application loads alert details from SQL Server.
  3. Analyst submits approve or reject decision with optional notes.
  4. Application validates required decision input and updates alert status and resolution notes.
  5. Analyst returns to queue with updated alert state.

Workflow 3: Fraud rule maintenance

  1. Analyst opens rules list to review existing rule definitions.
  2. Analyst creates, edits, or deletes a rule.
  3. Rule save validates mandatory values (rule name and severity), optional threshold/time-window formats, then persists changes.
  4. Updated rules list is displayed for continued maintenance.

Cross-Service Data Flows

The core data flow is intra-application between Struts actions and SQL Server tables, with one external composition path for authentication. During session resolution, the app may combine auth gateway output (sessionToken) with SQL user/token validation to form a usable SessionUser context. If the auth gateway call fails or returns non-200, login gracefully falls back to input state and protected workflows remain unavailable.

sequenceDiagram participant Analyst participant App as "ZavaFraudDetector" participant Session as "SsoSessionService" participant Auth as "Auth Gateway" participant DB as "SQL Server" Analyst->>App: Access flagged queue App->>Session: Resolve authenticated session alt Token already supplied Session->>DB: Validate token and active user DB-->>Session: Session user else Auth cookie only Session->>Auth: Resolve cookie to sessionToken alt Auth gateway available Auth-->>Session: sessionToken payload Session->>DB: Validate resolved token DB-->>Session: Session user else Auth gateway unavailable Note over Session: Authentication cannot be completed end end Session-->>App: SessionUser or null alt Session resolved App->>DB: Load new and in-review fraud alerts DB-->>App: Alert queue records Analyst->>App: Submit approve/reject decision App->>DB: Update alert status and notes DB-->>App: Persisted decision App-->>Analyst: Updated queue view else Session missing App-->>Analyst: Show login page end

Business Rules & Decision Logic

  • Protected workflows require a valid active session token linked to an active user.
  • Alert decision updates require both an alertId and non-empty decision; missing values result in no update.
  • Decision status is normalized to Approved or Rejected, and analyst username is appended to resolution notes for audit context.
  • Rule save enforces required fields (ruleName, severity) and numeric validation for threshold/time-window inputs before persistence.
  • All workflow persistence operations rely on synchronous SQL updates without asynchronous compensation flows.
