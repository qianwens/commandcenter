Data Storage & External Services

The application uses SQL Server (ZavaBankDB) as the primary system of record for users, session tokens, accounts, and transaction history. It integrates with an external auth gateway endpoint to resolve SSO tokens and with a ledger service endpoint to submit payment transactions.

Key Architectural Decisions

  • Uses a server-rendered Struts 1.x MVC pattern with synchronous request handling.
  • Keeps persistence logic directly in action/service classes through JDBC instead of a repository abstraction.
  • Uses token validation against both SQL data and a fallback auth gateway call for cross-platform SSO.

DTOs & Contracts

The API contract uses LoginForm and PaymentForm as request DTOs and returns server-rendered JSP views rather than JSON DTO responses. Supporting contract models include SessionUser (session identity), AccountOption (payment account selection), and PaymentHistoryRecord (history row projection). DTOs are mutable Java classes with getter/setter patterns; no OpenAPI, GraphQL, or protobuf contracts are present, and serialization is manual (XML payload assembled in PaymentAction for ledger calls).

Communication Patterns

Communication is synchronous throughout: browser to gateway over HTTP, gateway to SQL Server over JDBC, and gateway to external services over HTTP (WhoAmI.ashx and /api/transactions). No async messaging, service discovery, or circuit-breaker library is configured. Resilience is limited to connection/read timeouts (3s for auth gateway, 8s for ledger) and fallback behavior (failed auth/ledger calls return login or error message). Startup dependency chain affects availability: SQL Server, auth gateway, and ledger endpoint must be reachable for full behavior. Security posture: no TLS/auth enforcement is configured in this codebase; token-based session checks exist at application level without explicit transport security configuration.

Primary Workflows

Workflow 1: Session Login and Token Resolution

  1. User opens /login.do and submits token (or relies on cookie/header).
  2. LoginAction delegates token resolution to SsoSessionService.
  3. Service validates active token in SQL tables.
  4. If needed, service calls auth gateway WhoAmI.ashx to derive token from auth cookie.
  5. Successful resolution stores SessionUser in HTTP session and redirects to payment page.

Workflow 2: Post Payment to Ledger

  1. Authenticated user submits POST /makePayment.do with PaymentForm.
  2. PaymentAction validates required inputs and positive amount rule.
  3. Action reads account options and determines settlement account.
  4. Action sends XML transaction request to ledger API.
  5. On posted status, user is redirected to payment history; otherwise error is shown.

Workflow 3: View Payment History

  1. User opens /paymentHistory.do.
  2. PaymentHistoryAction ensures session identity is present.
  3. Action queries recent gateway-related transactions and maps rows.
  4. JSP renders transaction list or empty-state message.

Cross-Service Data Flows

The gateway composes business outcomes from local SQL data plus external service responses. Authentication may blend local SessionTokens data with auth gateway token resolution. Payment submission depends on ledger API acknowledgment; when ledger is unavailable, business flow degrades by keeping user on the payment page with an error rather than recording a posted payment.

Business Workflow Sequence

sequenceDiagram participant User participant Gateway as "ZavaPayGateway" participant DB as "SQL Server" participant Auth as "ZavaAuthGateway" participant Ledger as "ZavaLedger" User->>Gateway: Submit login token Gateway->>DB: Validate session token alt Token missing from DB Gateway->>Auth: Resolve token from auth cookie Auth-->>Gateway: Resolved session token Gateway->>DB: Recheck token validity end Gateway-->>User: Sign-in success and open payment page User->>Gateway: Submit payment request Gateway->>Gateway: Validate amount and required fields Gateway->>DB: Read account options and context Gateway->>Ledger: Post payment transaction alt Ledger accepted Ledger-->>Gateway: POSTED status Gateway-->>User: Redirect to history view else Ledger unavailable Note over Gateway: Show payment error and keep user on form Gateway-->>User: Payment not posted end User->>Gateway: Open payment history Gateway->>DB: Read recent posted transactions Gateway-->>User: Render history table

Business Rules & Decision Logic

  • Payment posting requires non-empty account, amount, and payment type, with amount greater than zero.
  • Session identity is mandatory for payment and history routes; unauthenticated requests are redirected to login flow.
  • Settlement account is adjusted to avoid debiting and crediting the same account in one transaction.
  • Business outcome depends on external ledger response containing POSTED; any failure results in non-posted payment feedback.

Caching Strategy

No explicit caching provider or cache abstraction is configured. Data is fetched directly from SQL Server per request, and user identity is cached only in HTTP session (SessionUser) after successful token resolution.

Data Ownership Boundaries

The solution is a single service with direct ownership over all queried relational tables in one SQL Server database. Cross-service data exchange occurs through HTTP integrations (auth gateway and ledger), while internal reads/writes happen through local JDBC queries. CQRS segregation is not implemented; read and write operations share the same database connection strategy.
