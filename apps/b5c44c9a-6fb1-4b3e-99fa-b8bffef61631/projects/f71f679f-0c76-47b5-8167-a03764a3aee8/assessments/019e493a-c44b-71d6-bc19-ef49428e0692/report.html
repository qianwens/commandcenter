Data Storage & External Services

The service relies on a SQL Server database (ZavaBankDB) accessed via ADO.NET with raw SQL queries. It reads from and writes to four tables: Transactions, Accounts, AccountAlerts, AlertRules, and AlertHistory. Connection details are resolved from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with a fallback to the web.config connection string. For messaging, the service publishes triggered alert events to a RabbitMQ exchange (zava.notifications) using the RabbitMQ Management HTTP API (not the AMQP protocol), with credentials and host resolved from environment variables.

Key Architectural Decisions

  • WCF SOAP service over basicHttpBinding: The service is exposed as a SOAP endpoint, making it a legacy-style integration point that requires WCF-compatible clients; no REST/JSON API is provided for core operations.
  • Direct ADO.NET without ORM: All database interactions use raw SqlConnection/SqlCommand with inline SQL strings rather than an ORM (e.g., Entity Framework), keeping the data access layer simple but tightly coupled to the SQL Server schema.
  • RabbitMQ via Management HTTP API: Instead of using the native AMQP client library, alert events are published via the RabbitMQ HTTP management API, avoiding a NuGet dependency but coupling the publisher to RabbitMQ's management port.

Communication Patterns

Synchronous (inbound):

Clients connect via WCF basicHttpBinding (HTTP transport, SOAP 1.1 envelope). There is no REST endpoint; all business operations require a SOAP-capable client or a WCF proxy generated from the WSDL.

Asynchronous (outbound — RabbitMQ):

When an alert rule is triggered, AlertService calls RabbitMqPublisher.PublishAlert(), which posts the alert event as a JSON payload to the RabbitMQ Management HTTP API (POST /api/exchanges/{vhost}/zava.notifications/publish). This is a fire-and-forget HTTP call with a boolean return (routed: true/false). There is no AMQP client library, no message acknowledgement, no dead-letter queue, and no retry on publish failure — if the HTTP call throws, the exception is silently caught and false is returned.

Resilience patterns:

None. There are no circuit breakers, retry policies, or bulkhead patterns (no Polly, Resilience4j, or equivalent). Database failures propagate as unhandled SqlException, and RabbitMQ publish failures are swallowed silently.

Service discovery:

No service discovery mechanism (Consul, Eureka, Kubernetes DNS). The RabbitMQ host/port and SQL Server host/port are resolved from environment variables at runtime with hardcoded defaults.

API gateway:

No API gateway is present. Clients call the WCF service directly.

Security posture:

No authentication or authorization is configured at the API or transport level. The WCF endpoint uses basicHttpBinding without any <security> element, meaning it operates in None security mode — all SOAP operations are publicly accessible with no credentials required. The health endpoint (/health) explicitly allows all users (<allow users="*"/>). There is no TLS/HTTPS configured, no JWT, no OAuth2, and no role-based access control. All endpoints are publicly accessible with no authorization checks.

Core Business Workflows

ZavaAlertService is the alert evaluation engine for Zava Bank: it checks every bank transaction against a set of configurable per-account alert rules and, when rules are triggered, publishes real-time notification events so that customers can be informed of significant activity on their accounts.

AlertService uses a shared database pattern: it reads Transactions and Accounts tables owned by other services rather than calling those services through APIs. The only outbound integration boundary is the RabbitMQ message exchange.

Primary Workflows

Workflow 1: Transaction Alert Evaluation

Entry point: EvaluateTransaction(transactionId) — called by an external orchestrator (e.g., the transaction processing service) after a transaction is committed.

Steps:

  1. Input validation: Reject transactionId ≤ 0 immediately with a SOAP fault.
  2. Transaction lookup: Query Transactions JOIN Accounts to retrieve the transaction's amount, post-transaction balance, account ID, customer ID, and channel. Fault if the transaction is not found.
  3. Rule loading: Query AccountAlerts JOIN AlertRules for all enabled and active alert rules configured on this account. Each rule carries a code, a threshold (account-specific override or rule default), and a notification channel.
  4. Rule evaluation loop: For each loaded rule, call the rule evaluator (IsTriggered) with the transaction data:
    • LOW_BAL: balance after transaction is below threshold.
    • LARGE_TXN: absolute transaction amount is at or above threshold.
    • OVERDRAFT: balance after transaction is negative (threshold ignored).
    • INTL_WIRE: channel contains "WIRE" and absolute amount is at or above threshold.
    • DAILY_SPEND: cumulative negative transactions on the same calendar day is at or above threshold (requires a secondary SQL aggregate query).
    • Unknown rule codes: never trigger (default false).
  1. Alert event publish: For each triggered rule, publish an AlertEvent to RabbitMQ via the management HTTP API. Capture the publish success/failure boolean.
  2. History recording: For each triggered rule, insert a row into AlertHistory recording the trigger value, notification sent flag, and a formatted details string (rule code, account, balance, channel).
  3. Response: Return AlertEvaluationResponse with the list of triggered rule codes, an aggregated publish flag, and a human-readable message.

Workflow 2: Alert Rule Management (Create / Update)

Entry points: CreateAlertRule(rule) and UpdateAlertRule(ruleId, rule).

Create flow:

  1. Validate input: rule must not be null; RuleName and RuleCode must be non-empty; AccountId and CustomerId must be positive.
  2. Open a SQL transaction.
  3. Insert a new row into AlertRules (normalising RuleCode to uppercase, defaulting Category to "Transaction" and Threshold to "0" when blank).
  4. Insert a new row into AccountAlerts linking the rule to the account, with notification channel defaulting to "Email".
  5. Commit. Return AlertRuleMutationResponse with the new RuleId and AccountAlertId.

Update flow:

  1. Validate ruleId > 0 and non-empty RuleName/RuleCode.
  2. Open a SQL transaction.
  3. Update the AlertRules row; fault if no row matched (rule not found).
  4. If AccountId > 0, upsert the AccountAlerts row (update if existing, insert if not).
  5. Commit. Return AlertRuleMutationResponse.

Workflow 3: Alert Rule Query

Entry point: GetAlertRules(accountId).

  1. Validate accountId > 0.
  2. Query AccountAlerts JOIN AlertRules for all rules configured on the account (both enabled and disabled), ordered by rule name.
  3. Return AlertRulesResponse with the full list of AlertRuleInfo records including threshold overrides, notification channels, and enabled state.

Workflow 4: Health Check

Entry point: GET /health.

Returns OK as plain text. No business logic; confirms the process is alive. Does not verify SQL Server or RabbitMQ connectivity.

Cross-Service Data Flows

ZavaAlertService is not an API gateway and performs no response aggregation. Its cross-service data interactions are:

  1. Inbound data read (shared DB): AlertService reads Transactions and Accounts tables written by other Zava Bank services. This is a direct SQL join — no API call. If those tables are unavailable or empty, EvaluateTransaction returns a SOAP fault ("Transaction was not found").
  2. Outbound event publish (RabbitMQ): On each triggered alert, AlertService posts an AlertEvent JSON payload to the zava.notifications exchange via the RabbitMQ Management HTTP API. The downstream notification service (identity unknown) consumes these events. Fallback behavior: if the HTTP publish call throws any exception, the exception is silently caught and published = false is recorded in AlertHistory. Alert evaluation and history recording continue regardless of messaging failure; there is no retry and no dead-letter fallback — triggered alerts can be lost silently if RabbitMQ is unreachable.

Input Validation Rules

  • transactionId must be > 0 (EvaluateTransaction); fault immediately otherwise.
  • accountId must be > 0 (GetAlertRules); fault immediately otherwise.
  • ruleId must be > 0 (UpdateAlertRule); fault immediately otherwise.
  • AlertRuleDefinition.RuleName and RuleCode must be non-empty (CreateAlertRule, UpdateAlertRule).
  • AlertRuleDefinition.AccountId and CustomerId must be > 0 for CreateAlertRule (not required for UpdateAlertRule).

Default Value Rules

  • RuleCode is normalised to uppercase on create/update.
  • Category defaults to "Transaction" when blank.
  • Threshold defaults to "0" when blank.
  • NotificationChannel defaults to "Email" when blank.
  • Account-specific threshold overrides the rule's DefaultThreshold unless the account threshold is an empty string (COALESCE logic in SQL).

Transaction Management

  • CreateAlertRule and UpdateAlertRule use explicit ADO.NET SqlTransaction spanning two table writes (AlertRules + AccountAlerts). No Rollback() is called explicitly — relied on connection disposal on exception. No distributed transaction or saga pattern is used.

Error Handling

  • All validation failures raise System.ServiceModel.FaultException with a human-readable message; WCF translates these to SOAP faults returned to the caller.
  • RabbitMQ publish failures are silently caught — no exception propagates to the caller, no retry, no dead-letter. This is a potential silent data loss scenario.
  • No structured business exception hierarchy; all faults are plain FaultException<string>.

Audit Trail

  • Every triggered alert rule produces an AlertHistory row (append-only) with: the trigger value (amount), a details string (rule code, account, balance, channel), whether a notification was sent, and (if sent) the notification timestamp. This serves as the primary audit trail for alert activity.

Authorization

  • No authorization logic is implemented at any level. Any caller with network access to port 8080 can invoke all WCF operations, including rule creation, update, and transaction evaluation.

Version & Compatibility Risks

All dependencies are .NET Framework 4.8 built-in BCL assemblies — there are no third-party NuGet packages. However, this creates significant migration risk: System.ServiceModel (WCF) is not available in .NET Core or .NET 5+; migrating to modern .NET would require replacing WCF with gRPC (Grpc.AspNetCore), REST (ASP.NET Core Web API), or CoreWCF. System.Web (which hosts the WCF service and the health handler) is also Windows/IIS-only and has no equivalent in .NET Core. System.Web.Script.Serialization.JavaScriptSerializer (used in RabbitMqPublisher) is a legacy JSON serializer deprecated in favor of System.Text.Json or Newtonsoft.Json. System.Data.SqlClient is being superseded by Microsoft.Data.SqlClient, which is the actively maintained fork with modern TLS and AAD support.

Notable Observations

  • No third-party NuGet packages: The project has zero NuGet dependencies (packages.config is empty), which simplifies dependency management but means the application is entirely reliant on legacy .NET Framework BCL APIs that do not port to modern .NET.
  • JavaScriptSerializer is deprecated: System.Web.Script.Serialization.JavaScriptSerializer (used in RabbitMqPublisher for alert event serialization) has been deprecated since .NET Framework 4.5 in favor of System.Text.Json; this should be replaced during any modernization effort.
  • No RabbitMQ AMQP client library: Rather than using the standard RabbitMQ.Client NuGet package (AMQP protocol), the service uses raw HttpWebRequest to call the RabbitMQ Management HTTP API — an unusual and fragile pattern that bypasses connection pooling, channel multiplexing, and acknowledgement semantics.
  • WCF server-side hosting is Windows/IIS-only: System.ServiceModel in this context hosts a WCF service via IIS, which locks the runtime to Windows. The Dockerfile (if used) would require a Windows Server base image; running on Linux containers is not feasible without CoreWCF migration.

Caching Strategy

No caching layer is present. Every request issues fresh SQL queries to SQL Server with no in-process cache, no distributed cache (Redis, Memcached), and no query result caching. Frequently read reference data such as AlertRules is reloaded from the database on every EvaluateTransaction call.

Data Ownership Boundaries

ZavaAlertService uses a shared database model: it accesses a single SQL Server database (ZavaBankDB) that also contains Transactions and Accounts tables owned and written by other Zava Bank services. AlertService reads these tables directly via SQL JOIN, creating tight coupling at the database level (no API boundary between services).

Data access patterns:

  • Transactions and Accounts: read-only by AlertService; assumed to be written by a separate transaction processing service.
  • AlertRules and AccountAlerts: fully owned by AlertService (create, read, update).
  • AlertHistory: append-only by AlertService; not read back through the API.

There is no CQRS separation, no event sourcing, and no read replica. All reads and writes go to the same SQL Server instance.
