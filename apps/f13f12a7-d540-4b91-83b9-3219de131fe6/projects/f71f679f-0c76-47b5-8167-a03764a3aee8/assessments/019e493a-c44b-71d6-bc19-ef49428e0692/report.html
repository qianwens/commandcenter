Data Storage & External Services

The application persists loan application and decision records in SQL Server and synchronously calls three external services (KYC, risk, and ledger) over HTTP JSON to complete an approval flow.

Key Architectural Decisions

  • Uses servlet-based endpoints configured via web.xml instead of annotation-driven controllers.
  • Uses direct JDBC SQL statements for persistence and bootstrap schema creation.
  • Performs synchronous orchestration of KYC, risk, and ledger APIs inside loan submission.

DTOs & Contracts

The API contract is JSON-based and currently uses dynamic JSON objects rather than dedicated request or response DTO classes. LoanApplyServlet treats the incoming body as an ad-hoc request contract and returns a composed decision response object. LoanStatusServlet returns a status payload built from database columns. No OpenAPI specification, protobuf schema, or GraphQL schema was detected.

Communication Patterns

The loan submission flow is synchronous: the API receives a request, validates required inputs, writes an initial application row, then calls KYC, risk, and ledger APIs over HTTP in sequence. Each external call uses a 7000 ms connect/read timeout. No retry, circuit breaker, or service discovery mechanism was detected; downstream URLs are configured as fixed base URLs from properties or environment variables. Security posture at API-contract level: no authentication or authorization checks were detected in servlet code, and downstream service calls use HTTP (not HTTPS/TLS).

Primary Workflows

Workflow 1: Submit Loan Application

  1. Client submits POST /api/loans/apply with customer and loan details.
  2. API validates required inputs and creates a submitted loan application record.
  3. API requests KYC verification; non-pass status routes to manual review outcome.
  4. If KYC passes, API requests risk score and approval recommendation.
  5. If risk declines, application is marked declined with reason.
  6. If risk approves, API requests ledger account creation.
  7. API persists decision details and returns final status.

Workflow 2: Retrieve Loan Status

  1. Client requests GET /api/loans/{id}/status.
  2. API validates path format and numeric identifier.
  3. API queries loan record and returns current status details or not-found response.

Cross-Service Data Flows

During submission, the loan origination service composes decision state from multiple external domains: KYC status first, then risk result, then ledger provisioning response. The composed outcome is persisted to local loan tables and returned to the caller. If KYC fails or risk declines, the workflow degrades to review or decline without invoking later steps; if ledger creation fails, the service returns an approval-pending-ledger status.

Business Rules & Decision Logic

  • Required input rule: customerId, loanProductId, requestedAmount, and termMonths must be valid positive values.
  • KYC gate rule: only KYC PASS proceeds to risk scoring; otherwise status is set to review.
  • Risk rule: risk response or fallback score threshold determines approve vs decline.
  • Ledger completion rule: approved loans attempt ledger account creation; failure results in APPROVED_PENDING_LEDGER.
  • Transaction boundary: submission flow wraps DB writes in one JDBC transaction with rollback on SQL errors.
