Data Storage & External Services

The portal reads and writes SQL Server tables (LoanProducts, LoanApplications) through direct SQL statements. It also calls an external authentication gateway for login/logout redirects and posts loan application XML payloads to a loan-origination API.

Key Architectural Decisions

  • Uses server-side Web Forms page lifecycle and event handlers as the primary interaction model.
  • Uses direct ADO.NET SQL commands rather than repository or ORM abstraction.
  • Uses an external authentication gateway while still enforcing Forms Authentication in the portal.

DTOs & Contracts

The portal constructs and sends an XML loanApplication contract containing customerId, loanProductId, requestedAmount, and termMonths to the external loan-origination API. Internally, request contracts are Web Forms field values and command arguments rather than typed API DTO classes. No OpenAPI, Swagger, GraphQL, or protobuf contracts were detected.

Communication Patterns

Communication is mostly synchronous. Browser interactions post back to the same ASPX page, and server-side code performs synchronous SQL access plus synchronous HTTP POST to the external loan-origination endpoint. No async message broker pattern, circuit breaker, retry policy, or service discovery mechanism was found. Transport security and API authentication for external calls are not explicitly configured in code; external URLs are plain HTTP.

Primary Workflows

Workflow 1: User Authentication Redirect

A user requests access to the portal and is redirected to Login.aspx if unauthenticated. The login page sends the user to ZavaAuthGateway with a ReturnUrl, then the user comes back with an authenticated session to continue.

Workflow 2: Loan Application Wizard Submission

The user completes wizard steps for personal details, employment data, and loan details. On finish, the portal validates amount and term values, posts an XML payload to the Loan Origination API, inserts a submitted application record in SQL Server, and refreshes history.

Workflow 3: Loan Application History Review

The dashboard loads the latest application entries and allows selecting a row for review intent. This supports operational visibility into recent submissions.

Cross-Service Data Flows

The portal composes data from its local database for product and history views, while orchestration with external services occurs through redirects (auth gateway) and direct HTTP POST (loan API). If the loan API call fails, the current implementation suppresses the exception and still proceeds with local insert, creating a degraded but continuing workflow.

Business Rules & Decision Logic

  • Access to protected pages requires authenticated users; Login.aspx and Logout.aspx are explicitly allowed.
  • Loan amount and term must be numeric before submission proceeds.
  • Submission status is set to Submitted with timestamped audit-style fields on insert.
  • On loan API communication errors, processing continues locally because the exception is caught without rethrow.
