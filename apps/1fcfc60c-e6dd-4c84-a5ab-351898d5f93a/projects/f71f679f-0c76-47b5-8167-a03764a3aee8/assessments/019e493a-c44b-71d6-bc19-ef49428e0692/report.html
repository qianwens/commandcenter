Data Storage & External Services

The worker writes audit data to SQL Server tables (AuditLog, AuditActions) and integrates with RabbitMQ for inbound event processing. A dead-letter exchange is used for failed message handling when parsing or persistence fails.

Key Architectural Decisions

  • Uses pull-based BasicGet polling instead of push-based consumers, with a configurable poll interval.
  • Supports both legacy and modern AuditLog schemas by detecting ActionID at runtime.
  • Stores original payload as raw details to preserve full audit evidence.

DTOs & Contracts

  • AuditEvent is the core service-level contract model used after payload parsing and before DB persistence.
  • Input contract supports two payload shapes: JSON object keys (eventType, serviceName, userId, timestamp, correlationId) and equivalent XML elements.
  • No gateway-level aggregation DTOs are present.
  • Serialization is handled with JavaScriptSerializer for JSON and XDocument for XML.
  • No OpenAPI/Swagger, protobuf, or GraphQL schema definitions are present.

Communication Patterns

  • Synchronous: Local in-process calls (PollQueue -> Parse* -> SaveAuditEvent) and synchronous SQL operations.
  • Asynchronous: RabbitMQ queue polling with manual ack/nack semantics.
  • Resilience patterns: No explicit circuit breaker or retry policy; failures are routed by broker dead-lettering via BasicNack(..., requeue: false).
  • Service discovery: None; broker and database are configured by environment/app settings host values.
  • Startup dependency chain: Worker requires RabbitMQ and SQL Server to be reachable before successful processing.
  • Security posture: No API authentication/authorization or TLS enforcement is implemented in this worker.

Core Business Workflows

The application’s primary business role is to transform inbound audit events into durable audit records for compliance and traceability. It handles ingestion, normalization, schema-aware persistence, and failure routing.

Primary Workflows

Workflow 1: Process inbound audit message

  1. Worker polls RabbitMQ queue audit.events.
  2. On message receipt, it determines payload format (JSON/XML) and normalizes into AuditEvent.
  3. It evaluates target schema (legacy vs modern) by inspecting AuditLog columns.
  4. It persists the audit record to SQL Server and acknowledges the message.

Workflow 2: Handle invalid or failed audit processing

  1. Worker receives a malformed payload or hits a database exception.
  2. Processing is aborted for the current message.
  3. Message is negatively acknowledged with requeue=false, letting broker dead-letter policy capture the event.
  4. Worker resumes polling on next interval.

Cross-Service Data Flows

Data flows one-way from upstream producers into RabbitMQ, then into this worker, and finally into SQL Server as the system of record. No response payload is sent back to producers; business outcome is represented by ack success or dead-letter routing on failure.

Business Workflow Sequence

sequenceDiagram participant Producer as "Upstream Service" participant Broker as "RabbitMQ Queue" participant Worker as "ZavaAuditWorker" participant SQL as "SQL Server" participant DeadLetter as "Dead Letter Exchange" Producer-->>Broker: Publish audit activity event Worker->>Broker: Poll next message Broker-->>Worker: Deliver event payload Worker->>Worker: Validate payload is present Worker->>Worker: Parse JSON or XML to AuditEvent Worker->>Worker: Determine legacy or modern schema alt Persistence succeeds Worker->>SQL: Insert audit record SQL-->>Worker: Insert confirmed Worker->>Broker: Ack message else Parse or persistence fails Worker->>Broker: Nack message requeue false Broker-->>DeadLetter: Route failed event Note over Worker: Processing continues in next poll cycle end

Business Rules & Decision Logic

  • Empty payloads are rejected and treated as processing failures.
  • Payload format decision rule: XML payloads are parsed when body begins with <; otherwise JSON parser is used.
  • Schema decision rule: if AuditLog.ActionID exists, write legacy row format; otherwise write modern row format.
  • Action normalization rule: blank event types are coerced to UNKNOWN when creating AuditActions.
  • Error-handling rule: any exception in parse/persist path leads to dead-letter routing rather than requeue.
  • Authorization rules are not implemented in this worker; trust boundary is external to the process.
