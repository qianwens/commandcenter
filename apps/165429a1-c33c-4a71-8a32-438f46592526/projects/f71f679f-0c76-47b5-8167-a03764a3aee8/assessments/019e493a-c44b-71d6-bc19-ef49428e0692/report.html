Data Storage & External Services

The service uses a single SQL Server database (ZavaBankDB) for user identity, role membership, and session token persistence. No message broker, cache, or third-party API integration is declared in the codebase.

Key Architectural Decisions

  • Uses a single-service architecture with UI pages and API handlers in the same deployable unit.
  • Persists session tokens in SQL Server while also embedding the token reference in the Forms Authentication ticket.
  • Uses environment-variable-first database configuration with fallback to web.config connection strings.

The contract is lightweight and mostly manual JSON construction rather than strongly typed API DTO libraries. AuthUser and SessionValidationResult act as service-level models used by handlers and page logic. API responses are composed in handlers/pages as JSON payloads with stable keys; no OpenAPI/Swagger, protobuf, or GraphQL schema artifacts were found.

Communication Patterns

All communication is synchronous: browser/portal to ASP.NET endpoints over HTTP and application to SQL Server over ADO.NET. No async messaging or service discovery mechanism is present. Resilience patterns such as circuit breakers, retries, or configured request timeouts were not detected. Startup dependency is limited to database reachability for authentication operations. Security posture: Forms Authentication cookie handling is enabled, anonymous access is denied by default except explicitly allowed paths, and there is no TLS termination configuration in-repo (expected to be handled by hosting infrastructure).

Core Business Workflows

The application handles user authentication lifecycle workflows for sign-in, identity lookup, token validation, and sign-out to support a portal-style login experience.

Primary Workflows

Workflow 1: User Login and Session Issuance

  1. User submits credentials on Login.aspx.
  2. Service loads user identity and status from persistence.
  3. Password hash check is performed using stored salt and SHA1 comparison.
  4. On success, a session token is generated, persisted, and embedded into Forms Authentication ticket data.
  5. User is redirected to the post-login page.

Workflow 2: Session Validation for Downstream Consumers

  1. Client or downstream portal calls /api/auth/validate with token.
  2. Service validates token activity and expiry against session records.
  3. If valid, response includes user identity and expiration metadata.
  4. If invalid/missing, response indicates non-valid state.

Workflow 3: Current Identity Projection

  1. Client calls WhoAmI.ashx with forms auth cookie.
  2. Service decrypts ticket, reloads active user and roles, and derives display name.
  3. Service returns authenticated identity payload including role list and session token.

Cross-Service Data Flows

No multi-service composition or gateway fan-out is implemented in-repo. The gateway exchanges data only with its SQL Server backend and returns composed identity/session payloads directly to callers. There is no fallback path to alternate services when database lookups fail.

Business Rules & Decision Logic

  • Authentication succeeds only when user exists, is active, not locked out, and supplied password hash matches stored hash.
  • Session tokens are created with a 30-minute expiry and marked inactive during logout.
  • Token validation requires active token, non-expired timestamp, and active user account.
  • WhoAmI identity response is denied when auth cookie is absent, invalid, expired, or mapped to inactive/locked account.

Version & Compatibility Risks

The service targets .NET Framework 4.8, which is Windows-centric and commonly requires migration work for Linux-native hosting and modernization to .NET (Core/5+). The Dockerfile currently relies on Mono, which can differ behaviorally from IIS-hosted .NET Framework workloads.

Notable Observations

  • packages.config is empty, so the application mainly relies on framework assemblies.
  • Data access is implemented with raw SQL commands; there is no ORM abstraction layer.
  • Security behavior depends on Forms Authentication and static machine keys in web.config.
