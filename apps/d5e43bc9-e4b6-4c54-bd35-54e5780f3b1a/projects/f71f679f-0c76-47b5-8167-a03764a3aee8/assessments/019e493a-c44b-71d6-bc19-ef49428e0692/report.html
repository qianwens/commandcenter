Data Storage & External Services

The service primarily uses filesystem directories for operational state (incoming/processed/error), posts transaction records to an external ledger HTTP API, and publishes status events to RabbitMQ exchanges.

Key Architectural Decisions

  • Uses a polling loop over a shared directory instead of event-driven file watchers.
  • Implements integration boundaries through HTTP for ledger posting and AMQP topic publishing for downstream event consumers.
  • Keeps parsing, posting, and event publication in a single process entry point for operational simplicity.

DTOs & Contracts

The service constructs two contract payload types in-code: a ledger transaction request payload (source, fileName, batchNumber, routingNumber, accountNumber, amountCents, traceNumber) and event payloads for ach.processed and ach.failed. These are generated as JSON strings and function as service-level integration contracts. No separate OpenAPI, protobuf, or GraphQL schema files were found.

Communication Patterns

The worker uses synchronous HTTP POST calls to a configurable ledger endpoint for each ACH entry and asynchronous AMQP topic publishing for success/failure events. No service discovery or API gateway is used; integrations are URL/host configured. Retry/circuit breaker frameworks are not implemented; failures are handled by moving files to error path and publishing failure events. No explicit API-level authentication, authorization, or TLS enforcement is configured in the code paths inspected.

Core Business Workflows

This application processes inbound ACH files and coordinates transaction submission plus outcome signaling for downstream banking workflows.

Primary Workflows

Workflow 1: Process incoming ACH file

  1. Worker polls incoming directory for files.
  2. File parser extracts batch header, detail entries, and control records.
  3. For each ACH entry, worker submits transaction payload to ledger service.
  4. On successful post, worker emits ach.processed event.
  5. If all entries succeed, file is moved to processed directory.
  6. If parsing or posting fails, worker emits failure event and moves file to error directory.

Cross-Service Data Flows

The worker composes data from parsed ACH records and pushes it to two external boundaries: synchronous ledger transaction API and asynchronous RabbitMQ topics. When ledger posting is unavailable or returns non-success status, business outcome degrades to failed-file handling and dead-letter style failure notification.

Business Workflow Sequence

sequenceDiagram participant Ops as "Operations" participant Worker as "ZavaACHProcessor" participant Ledger as "Ledger Service" participant MQ as "RabbitMQ" Ops->>Worker: Place ACH file in incoming directory Worker->>Worker: Parse batch and entries Worker->>Ledger: Submit entry transaction alt Ledger accepted Ledger-->>Worker: Success response Worker->>MQ: Publish ach.processed Worker-->>Ops: Move file to processed else Ledger rejected or timeout Ledger-->>Worker: Error/timeout Worker->>MQ: Publish ach.failed Note over Worker: File marked as failed Worker-->>Ops: Move file to error end

Business Rules & Decision Logic

  • Files without valid ACH entries are considered parse failures and routed to error handling.
  • A file is considered fully successful only when all entry posts succeed.
  • Any failed post in a batch causes the entire file to be routed to error path.
  • Outcome events are emitted to support downstream monitoring and reconciliation processes.
