Data Storage & External Services

The application uses a single Microsoft SQL Server database table (Documents) to store document metadata and file bytes (VARBINARY(MAX)). No cache, queue, or third-party API integration is defined in the current codebase.

Key Architectural Decisions

  • Uses a simple servlet-per-use-case pattern rather than a layered Spring-style framework.
  • Creates the Documents table on startup via DocVaultBootstrapServlet to reduce external migration dependencies.
  • Reads database configuration from environment variables first, with doc-vault.properties as fallback.

DTOs & Contracts

The API contract is servlet-native and uses lightweight JSON payloads built with org.json rather than dedicated DTO classes. Request/response contract roles include multipart upload fields (customerId, documentType, file) and JSON response objects for upload/list operations. No OpenAPI, protobuf, or GraphQL contract definitions are present.

Communication Patterns

All application communication is synchronous. Clients call servlet endpoints over HTTP; servlets execute SQL queries via JDBC and return immediate responses. No async messaging, service discovery, gateway aggregation, circuit breaker, retry, or timeout policy abstraction is configured in code. Security posture is minimal: no API authentication, no authorization checks, and no TLS enforcement in the application layer.

Core Business Workflows

ZavaDocVault supports a banking document-management workflow where staff upload customer documents and retrieve them later for review or download. The domain centers on customer-linked documents and their lifecycle from intake to retrieval.

Primary Workflows

Workflow 1: Upload customer document

Entry point: POST /api/documents/upload.

  1. Staff submits customer ID, optional document type, and a file.
  2. Service validates customer ID format and ensures a non-empty file exists.
  3. Service sanitizes file name and stores metadata plus binary payload.
  4. Service returns upload confirmation with generated document ID.

Business rules involved: customer ID must be numeric; file is required; document type defaults to General when omitted.

Workflow 2: Retrieve customer documents

Entry point: GET /api/documents/customer/{customerId}.

  1. User requests all documents for a customer.
  2. Service validates customer ID format.
  3. Service queries documents ordered by most recent upload.
  4. Service returns a JSON list with count and metadata.

Business rules involved: customer ID must be numeric; results sorted by upload date and document ID descending.

Workflow 3: Download specific document

Entry point: GET /api/documents/{id}.

  1. User requests a document by ID.
  2. Service validates ID format and queries storage.
  3. If found, service streams file bytes with attachment headers.
  4. If not found, service returns an explicit not-found response.

Business rules involved: document ID must be numeric; document must exist to download.

Cross-Service Data Flows

No multi-service composition is implemented. Workflow data flows between a single web service and SQL Server; therefore no gateway aggregation or cross-service fallback path exists in the current system.

Business Workflow Sequence

sequenceDiagram participant Staff participant App as "ZavaDocVault Service" participant DB as "SQL Server" Staff->>App: Submit upload request with file App->>App: Validate customer id and required file App->>DB: Save document metadata and binary content DB-->>App: Persisted with document id App-->>Staff: Upload confirmed Staff->>App: Request customer documents App->>App: Validate customer id App->>DB: Query documents by customer id DB-->>App: Return ordered document rows App-->>Staff: JSON document list Staff->>App: Request document download by id App->>DB: Query file payload by document id alt Document exists DB-->>App: File metadata and bytes App-->>Staff: File download response else Document missing DB-->>App: No record found App-->>Staff: Not found error response end

Business Rules & Decision Logic

  • Validation rules: customerId and document ID values must be numeric; uploaded file must be present and non-empty.
  • Decision logic: documentType defaults to General when omitted; missing document ID yields not-found response.
  • Data integrity rules: every document is linked to a customer ID; uploaded documents are timestamped at insert time.
  • Transactions and errors: each workflow step performs single JDBC statements and returns explicit HTTP error responses for invalid input or persistence failures.
  • Authorization: no role-based or ownership authorization checks are implemented at business logic level.
