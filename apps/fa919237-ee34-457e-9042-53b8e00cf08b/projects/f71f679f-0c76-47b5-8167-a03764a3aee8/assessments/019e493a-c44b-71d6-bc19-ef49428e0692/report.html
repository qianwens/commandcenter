Data Storage & External Services

The application uses Microsoft SQL Server as its primary data store, accessed via raw JDBC (no ORM), to persist wire transfer records including amounts, currency codes, FX rates, statuses, and audit timestamps. An external Currency Service (HTTP REST) is called for international transfers to retrieve live FX rates. A Ledger Service (HTTP REST, XML payload) is called to post debit transactions against the originating account. RabbitMQ is used as an async message broker where each wire transfer completion (or failure) publishes an event to a configurable topic exchange with a routing key derived from the transfer status.

Key Architectural Decisions

  • Direct JDBC over ORM: Data access uses raw DriverManager connections and PreparedStatement without a JPA/ORM layer, keeping the footprint minimal but coupling data access logic directly into servlet classes.
  • No DI framework: There is no dependency injection container; WireConfig acts as a static configuration registry reading values from environment variables with property-file fallback.
  • Schema bootstrap on startup: WireBootstrapServlet runs at load-on-startup=1 to create the WireTransfers table if it does not exist, making the service self-initializing against an empty database.

DTOs & Contracts

No formal DTO or model classes exist. All request parsing and response serialization are performed inline within the servlet methods using org.json.JSONObject.

POST /api/wire/transfer — request fields (parsed inline):

fromAccountId (int), toAccountNumber (string), toRoutingNumber (string), beneficiaryName (string), currencyCode (string, default "USD"), amount (double). See data-architecture.md for full field definitions and storage schema.

POST /api/wire/transfer — response fields:

wireTransferId (long), status (string: "POSTED" or "PENDING_LEDGER"), referenceNumber (string, format "WIRE-XXXXXXXX"), fxRate (BigDecimal), debitAmountUSD (BigDecimal), ledgerPosted (boolean).

GET /api/wire/{id}/status — response fields:

wireTransferId, status, referenceNumber, amount, currencyCode, fxRate, debitAmountUSD, isInternational, failureReason, createdDate, updatedDate.

No DTOs are immutable (no Java records, no Lombok). No OpenAPI/Swagger specification, no Protobuf or GraphQL schemas. Serialization uses org.json.JSONObject.toString() directly; no Jackson or Gson configuration.

Communication Patterns

Synchronous outbound (HTTP):

Two blocking HTTP calls are made from WireTransferServlet using java.net.HttpURLConnection (no RestTemplate, WebClient, or Feign):

  • Currency Service — GET {CURRENCY_SERVICE_BASE_URL}/api/currency/rate?from=USD&to={currency} — connect timeout: 7 000 ms, read timeout: 7 000 ms. On any failure (exception or non-2xx), fxRate silently falls back to 1.0.
  • Ledger Service — POST {LEDGER_SERVICE_BASE_URL}/api/transactions — XML body — connect timeout: 7 000 ms, read timeout: 7 000 ms. On failure, the wire transfer is persisted with status="PENDING_LEDGER" and ledgerPosted=false rather than aborting the request.

Asynchronous outbound (AMQP):

After database insertion, WireTransferServlet.publishWireEvent() opens an AMQP connection to RabbitMQ, declares a durable topic exchange (rabbitmq.exchange), and publishes a JSON event with routing key {prefix}.{status_lowercase}. This is fire-and-forget: any AMQP exception is silently swallowed and does not affect the HTTP response to the caller.

Resilience patterns:

No circuit breaker, retry policy, bulkhead, or rate limiter is implemented. The 7-second HTTP timeout on downstream calls is the only resilience measure. A slow or unresponsive Currency Service or Ledger Service will block the servlet thread for up to 7 seconds per request.

Service discovery:

All downstream URLs are hardcoded via environment variables (CURRENCY_SERVICE_BASE_URL, LEDGER_SERVICE_BASE_URL, RABBITMQ_HOST). No service registry (Eureka, Consul, Kubernetes DNS-based discovery) is in use.

API gateway:

None. The service is directly exposed on port 8080 with no reverse proxy, API gateway, or BFF layer in the declared configuration.

Startup dependency chain:

WireBootstrapServlet runs at load-on-startup=1 and attempts a SQL Server connection on startup. If the database is unavailable, the servlet container will fail to initialize and the service will not start. For full probe configuration details, see configuration-inventory.md.

Security posture:

No authentication, no authorization, and no TLS are configured. All endpoints (/api/wire/transfer, /api/wire/{id}/status, /health) are publicly accessible with no credentials, tokens, or role checks required. Outbound HTTP calls to the Currency Service and Ledger Service also use plain HTTP (no HTTPS enforcement). This represents a critical security gap for a financial transaction service.

Core Business Workflows

ZavaWireTransferService is a wire transfer processing engine that accepts transfer requests from banking clients, validates them, applies foreign exchange conversion for international payments, posts debit entries to a ledger, records the transaction, and broadcasts transfer events for downstream processing.

Primary Workflows

Workflow 1: Initiate Wire Transfer

Entry point: POST /api/wire/transfer

This is the core business process. The servlet receives a JSON transfer request, validates it, determines whether FX conversion is needed, coordinates with external services, persists the record, and publishes a status event.

Steps:

  1. Parse request body — Read raw HTTP body and deserialize to a JSONObject. On parse failure → reject with 400.
  2. Extract and validate required fields — fromAccountId, toAccountNumber, toRoutingNumber, beneficiaryName, and amount are all required and must be non-empty/positive. currencyCode defaults to "USD" if absent.
  3. Validate ABA routing number — Apply the ABA checksum algorithm (weights 3-7-1-3-7-1-3-7-1, sum mod 10 must be 0, exactly 9 digits). Reject with 400 if invalid.
  4. Determine transfer type — If currencyCode != "USD", the transfer is flagged as international.
  5. Obtain FX rate (international only) — Synchronously call the Currency Service. If the call fails or times out (7 s), default fxRate to 1.0 (no conversion).
  6. Calculate debit amount — debitAmountUSD = amount × fxRate, rounded to 2 decimal places (HALF_UP).
  7. Generate reference number — Format: "WIRE-" + first 8 characters of a random UUID (uppercased).
  8. Post ledger debit — Synchronously POST an XML debit instruction to the Ledger Service. If the call succeeds (2xx), mark status = "POSTED"; otherwise mark status = "PENDING_LEDGER" with failureReason = "Ledger posting failed.".
  9. Persist wire transfer — INSERT into WireTransfers table. On SQL failure → respond 500 (the ledger debit already posted — this is a non-atomic gap).
  10. Publish wire event — Publish a JSON event to RabbitMQ topic exchange with routing key {prefix}.{status_lowercase}. Failure is silently swallowed.
  11. Respond to caller — Return 200 JSON with wireTransferId, status, referenceNumber, fxRate, debitAmountUSD, ledgerPosted.

Workflow 2: Query Wire Transfer Status

Entry point: GET /api/wire/{id}/status

A read-only workflow for retrieving the current state of a previously submitted wire transfer.

Steps:

  1. Extract and validate ID — Parse the numeric wireTransferId from the URL path segment. Reject with 400 if not numeric or path is malformed.
  2. Fetch record from database — Query WireTransfers by primary key.
  3. Return status payload — Return 200 JSON with all stored fields (status, amounts, dates, failure reason). Return 404 if not found.

Workflow 3: Schema Initialization (Startup)

Entry point: WireBootstrapServlet.init() — triggered at load-on-startup=1.

Steps:

  1. Open database connection — Attempt JDBC connection on Tomcat startup.
  2. Create table if absent — Execute IF OBJECT_ID('WireTransfers', 'U') IS NULL CREATE TABLE WireTransfers (...).
  3. Fail fast if DB unavailable — Any SQL failure throws ServletException, preventing the application from serving requests.

Cross-Service Data Flows

Wire Transfer submission is the only cross-service workflow:

  1. FX rate (conditional): WireTransferServlet calls the Currency Service synchronously with GET /api/currency/rate?from=USD&to={currencyCode}. The response provides the rate field (double). If unavailable, fxRate silently defaults to 1.0 — meaning an international transfer may be processed at the wrong exchange rate without any alert.
  2. Ledger debit: WireTransferServlet calls the Ledger Service synchronously with POST /api/transactions (XML body containing debitAccountId, creditAccountId, amount, referenceNumber). Success determines the wire status. There is no compensation mechanism: if the ledger POST succeeds but the subsequent database INSERT fails, the debit is recorded in the ledger but no WireTransfer record exists in this service.
  3. Event publishing: After a successful database INSERT, a JSON event is published to RabbitMQ. Downstream consumers (not part of this service) are expected to react to wire.posted or wire.pending_ledger routing keys. Publish failures do not affect the HTTP response.

Transactions & Data Integrity

  • No transaction management: There is no @Transactional scope or explicit JDBC transaction wrapping. The ledger debit and database insert are not atomic. A successful ledger POST followed by a database INSERT failure leaves a debit with no corresponding wire transfer record — a financial reconciliation problem.
  • No compensation: There is no retry mechanism, saga orchestration, or compensating transaction to reverse a ledger debit in case of DB failure.

Version & Compatibility Risks

javax.servlet:javax.servlet-api:4.0.1 targets the legacy javax.servlet namespace (Jakarta EE 8 and below). From Jakarta EE 9 onward the package was renamed to jakarta.servlet, and Tomcat 10+ requires this new namespace. Any upgrade from Tomcat 9 to Tomcat 10/11 (needed for modern Java 17/21 LTS alignment) would require migrating all import statements across the codebase. org.json:json:20140107 is pinned to a 2014 release — over a decade old — with many patch and feature versions now available (e.g., 20240303); while the API is stable, the old version carries potential JSON-parsing edge-case bugs. com.microsoft.sqlserver:mssql-jdbc:12.6.3.jre11 and com.rabbitmq:amqp-client:5.20.0 are current, well-maintained releases with no immediate compatibility concerns.

Notable Observations

  • Minimal dependency footprint: Only 4 declared dependencies with no BOM or dependency management block. This simplifies upgrades but means no centralized version governance.

Caching Strategy

No caching layer is present. Every request to /api/wire/{id}/status issues a live SQL query to SQL Server. There are no in-memory caches, distributed cache providers (Redis, EhCache, etc.), query result caches, or second-level ORM caches. Wire transfer data is write-once (inserted and never updated by this service) but no read caching has been introduced.

Data Ownership Boundaries

ZavaWireTransferService is the sole owner of the WireTransfers table. No other service directly accesses this table (there is no evidence of shared schema access). The FromAccountID field references an account in an external system (the Ledger Service owns account records), but this is a logical foreign key only — no referential integrity constraint is declared in the database.

Cross-service data access is exclusively via HTTP APIs:

  • Account balance/debit is delegated to the Ledger Service via REST (POST XML).
  • FX rates are fetched from the Currency Service via REST (GET JSON).
  • Wire events are published to RabbitMQ for downstream consumers.

No CQRS pattern, event sourcing, or read-model separation is in place. The single WireTransfers table serves both the write path (POST /api/wire/transfer) and the read path (GET /api/wire/{id}/status).
