Data Storage & External Services

The application writes execution status records to SQL Server (BatchRunLog) and coordinates three external HTTP batch services for interest accrual, statement generation, and reconciliation.

Key Architectural Decisions

  • Uses a single-threaded scheduled executor to enforce sequential batch execution.
  • Keeps integration lightweight with direct HttpURLConnection calls and JSON payloads.
  • Persists only run-level audit results instead of full domain data in this service.

Primary Workflows

Workflow 1: Nightly Batch Run Orchestration

  1. Scheduler wakes up based on configured interval.
  2. Service invokes interest accrual endpoint.
  3. Service invokes statement generation endpoint.
  4. Service invokes daily reconciliation endpoint.
  5. Service computes overall run result (SUCCESS only if all calls succeed).
  6. Service writes a BatchRunLog record with status and aggregated details.

Workflow 2: Graceful Shutdown Handling

  1. Runtime shutdown hook marks scheduler as stopped.
  2. Active scheduler thread is interrupted.
  3. Application exits after loop observes stopped state.

Cross-Service Data Flows

The scheduler composes a run-level outcome by collecting HTTP success/failure results from three downstream services and then persisting the aggregated status to SQL Server. If one downstream service fails, the workflow degrades to a failed batch result while still attempting to complete and log the run summary.

Business Workflow Sequence

sequenceDiagram participant Timer as "Scheduler Timer" participant Scheduler as "ZavaBatchScheduler" participant Interest as "Interest Service" participant Statement as "Statement Service" participant Ledger as "Ledger Service" participant LogDB as "BatchRunLog Store" Timer->>Scheduler: Trigger scheduled batch run Scheduler->>Interest: Execute interest accrual job Interest-->>Scheduler: Success or failure Scheduler->>Statement: Execute statement generation job Statement-->>Scheduler: Success or failure Scheduler->>Ledger: Execute reconciliation job Ledger-->>Scheduler: Success or failure alt All jobs succeeded Scheduler->>Scheduler: Mark status SUCCESS else One or more jobs failed Note over Scheduler: Mark status FAILED with per-job details end Scheduler->>LogDB: Persist BatchRunLog(runType, startedAt, endedAt, status, details) LogDB-->>Scheduler: Write complete

Business Rules & Decision Logic

  • Overall run status is SUCCESS only when all three downstream calls return HTTP 2xx.
  • Missing endpoint/base URL is treated as a failed job path.
  • Each run writes an audit record regardless of success/failure of downstream operations.
  • Scheduler interval defaults to 300000 ms and can be overridden via environment variable.
