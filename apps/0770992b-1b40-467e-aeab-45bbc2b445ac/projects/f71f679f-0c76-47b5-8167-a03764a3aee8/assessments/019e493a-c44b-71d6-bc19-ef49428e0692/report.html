Cloud Readiness

Issue CategoryCriticalityStory Point
SQL Server connection has TLS encryption disabled
 Mandatory 1
FilePosition
src/main/java/com/zavabank/kycservice/KycConfig.javaLine 28

Explanation

The JDBC URL is constructed with encrypt=false;trustServerCertificate=true, which disables TLS encryption for all traffic between the application and the SQL Server instance. In a cloud deployment, database traffic typically traverses shared network infrastructure. Unencrypted connections expose query data, credentials, and result sets to network-level interception. This must be remediated before production deployment: set encrypt=true and configure a trusted certificate.
Database credentials stored in application bundle

Explanation

The file src/main/resources/kyc.properties contains database host, username, and a plaintext password that are bundled inside the WAR artifact and committed to source control. Any person with repository read access or access to the deployed WAR file can extract these credentials. Cloud-native deployments must source secrets from a secrets manager (Azure Key Vault, AWS Secrets Manager, Kubernetes Secrets) and never commit credentials to source control. Data Storage & External Services

The application uses a single Microsoft SQL Server instance (ZavaBankDB) as its only data store. Two tables are managed: KYCWatchlist (holds names flagged for review, e.g. from OFAC sample lists) and KYCVerification (audit log of every verification attempt). Connection parameters (host, port, database name, credentials) are supplied via environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with a fallback to the bundled kyc.properties resource file.

Key Architectural Decisions

  • Direct JDBC access — the application uses raw PreparedStatement calls rather than an ORM, keeping the footprint minimal but placing schema bootstrap and SQL logic directly inside servlet methods.
  • Servlet-per-endpoint pattern — each HTTP endpoint is an individual HttpServlet subclass registered in web.xml, following classic Jakarta EE conventions rather than a front-controller or REST framework.
  • Configuration via environment variables with property-file fallback — KycConfig first reads environment variables, falling back to kyc.properties, enabling both container-native injection and local development use.

DTOs & Contracts

The application has no formal DTO or model classes. All request and response payloads are parsed and assembled inline inside each servlet using org.json.JSONObject:

  • Verify request (POST /api/kyc/verify): raw JSON body read via HttpServletRequest.getReader(); fields extracted with optInt/optString — no schema validation or binding framework.
  • Verify response: JSONObject built in-memory with keys customerId, status (PASS or REVIEW), and reason.
  • Status response: JSONObject built from a JDBC ResultSet with keys customerId, status, reason, fullName, idNumber, and verifiedDate.
  • Error responses: Hard-coded inline JSON string literals (e.g. "{\"status\":\"ERROR\",\"message\":\"...\"}") written directly to the response writer.

There are no OpenAPI/Swagger specifications, protobuf schemas, or GraphQL schemas. No Jackson, Gson, or other serialization configuration is used beyond the basic org.json library.

Communication Patterns

Synchronous only. All inter-layer communication is synchronous:

  • HTTP clients call the Servlet endpoints directly over HTTP.
  • Each servlet opens a raw JDBC connection to SQL Server, executes queries, and closes the connection — there is no asynchronous messaging, event bus, or message broker.

No resilience patterns. There is no circuit breaker, retry policy, timeout configuration, or connection pool. Each request acquires a raw java.sql.Connection via DriverManager.getConnection() and closes it after the request. This means every API call is one JDBC open-and-close cycle; under concurrent load, connection exhaustion is a significant risk.

No service discovery or API gateway. The service is a self-contained WAR deployed directly to Tomcat. All routing is handled by web.xml servlet mappings.

Security posture: none. There is no authentication (no JWT, OAuth2, Basic Auth, session management), no authorization (no role checks, no @RolesAllowed), and no TLS configuration. All four URL patterns are publicly accessible with no authorization checks whatsoever. The /internal/bootstrap path is exposed with no access restriction.

Startup dependency chain. KycBootstrapServlet is registered with <load-on-startup>1</load-on-startup> in web.xml, ensuring the DDL initialization runs before the service begins serving traffic. If the SQL Server connection fails during startup, Tomcat will throw a ServletException and the application will fail to deploy, which acts as an implicit readiness gate.

Core Business Workflows

ZavaKYCService automates the Know Your Customer (KYC) identity-screening process by checking customer names against a sanctions/watchlist and recording the verification outcome for compliance audit purposes.

Domain Entities

Each entry holds a person's full name and the source list that flagged them.Logically referenced by every verification attempt — if a customer name matches a watchlist entry, the verification is flagged for review
KYCVerificationKYC ScreeningAn immutable audit record of each customer identity-check attempt. Service-to-Domain Mapping

ServiceDomain ContextOwned EntitiesExternal Dependencies

This is a single-service application. There are no cross-service data flows; the service is the exclusive source of truth for both entities.

Primary Workflows

Workflow 1: Application Startup — Schema Bootstrap

Triggered automatically when the Tomcat servlet container starts, before any HTTP traffic is accepted.

  1. KycBootstrapServlet.init() is invoked by Tomcat (declared load-on-startup=1 in web.xml).
  2. A JDBC connection is opened to ZavaBankDB.
  3. An idempotent DDL statement checks whether KYCWatchlist exists; if not, it is created with the required columns.
  4. The same statement checks whether KYCVerification exists; if not, it is created.
  5. If KYCWatchlist has no rows, a single seed entry (John Alias / OFAC-SAMPLE) is inserted to bootstrap the watchlist for development and testing purposes.
  6. If any step fails (SQL Server unreachable, DDL error), a ServletException is thrown and Tomcat aborts the deployment — no partial startup is possible.

Business significance: Guarantees that the database schema is always in sync with application expectations before service traffic begins. The seed watchlist entry allows immediate functional testing of the REVIEW path.

Workflow 2: KYC Verification — Name Screening

The primary business workflow. A calling system (e.g., a customer onboarding service) submits a customer's details for identity screening.

  1. Caller sends POST /api/kyc/verify with a JSON body containing customerId, fullName, and optionally idNumber.
  2. Input validation:
    • The JSON body must be parseable; an unparseable body returns HTTP 400.
    • customerId must be a positive integer (≥ 1); zero or missing returns HTTP 400.
    • fullName must be present and non-empty after trimming; missing or blank returns HTTP 400.
  1. A JDBC connection is opened.
  2. Watchlist check: the supplied fullName is compared case-insensitively (UPPER(FullName) = UPPER(?)) against all entries in KYCWatchlist.
  3. Status determination:
    • If a match is found → status = REVIEW, reason = "Name matched watchlist."
    • If no match is found → status = PASS, reason = "No watchlist match."
  1. The outcome is written to KYCVerification with the CustomerID, FullName, IdNumber, computed Status, Reason, and the current timestamp.
  2. The response JSON {customerId, status, reason} is returned with HTTP 200.
  3. On any SQL error, HTTP 500 is returned with a generic error message; no partial record is guaranteed.

Workflow 3: KYC Status Retrieval

Allows a caller to query the most recent screening outcome for a known customer.

  1. Caller sends GET /api/kyc/status/{customerId}.
  2. Input validation: customerId path segment must be a numeric integer; non-numeric returns HTTP 400.
  3. A JDBC connection is opened.
  4. The most recent KYCVerification record for the given CustomerID is retrieved, ordered by VerifiedDate DESC then VerificationID DESC.
  5. If no record exists → HTTP 404 with {customerId, status: "NOT_FOUND"}.
  6. If a record exists → HTTP 200 with {customerId, status, reason, fullName, idNumber, verifiedDate}.
  7. On SQL error → HTTP 500 with a generic error message.

Cross-Service Data Flows

ZavaKYCService is a standalone service with no upstream or downstream service dependencies. All data flows are between the HTTP client (caller) and the SQL Server database via the application's JDBC layer. There is no service aggregation, event publishing, or consumption pattern.

If the SQL Server instance is unavailable during request handling (Workflows 2 and 3), the JDBC connection attempt throws a SQLException which is caught and translated to an HTTP 500 response. There is no circuit breaker, retry, or graceful degradation — the service is entirely dependent on SQL Server availability for every request.

Decision Logic

Watchlist name matching: The sole business decision in the service. The customer's fullName is compared against KYCWatchlist.FullName using a case-insensitive exact match (UPPER(FullName) = UPPER(?)). There is no fuzzy matching, phonetic matching, or partial-name matching — only exact (case-insensitive) full-name equality triggers a REVIEW status. This means minor spelling variations or middle-name differences would result in a PASS even if the underlying identity is a watchlist match.

State Transitions

KYC verification results have two terminal states with no lifecycle transitions:


(verification submitted) → PASS  (no match found — cleared)
(verification submitted) → REVIEW (match found — requires manual review)

There is no REJECTED, APPROVED, PENDING, or CLEARED follow-up state tracked within this service. Each call to /api/kyc/verify creates a new independent record; there is no update/override mechanism.

Business Constraints

  • No idempotency: Re-submitting the same customerId + fullName pair creates a new KYCVerification row each time. There is no deduplication or update-in-place logic.
  • Audit completeness: Every submitted verification (PASS or REVIEW) is persisted, providing a complete audit trail. However, failed requests (HTTP 400/500) produce no audit record.
  • Watchlist is static: There is no API to add, remove, or update watchlist entries. The watchlist can only be managed directly via database DML. The only automated write to KYCWatchlist is the single seed entry inserted at bootstrap.

Transactions

No explicit transaction management is configured. Each JDBC operation within a request handler uses the connection's default auto-commit mode. The watchlist SELECT and verification INSERT in Workflow 2 are executed on the same connection but in separate statements — there is no explicit transaction wrapping them, meaning a failure between the two statements could leave the verification unrecorded without any rollback or retry.

Error Handling

All SQLException exceptions in the request path are caught and translated to HTTP 500 with a generic message ("Unable to complete KYC verification." or "Unable to fetch KYC status."). Exceptions are not logged — there is no audit trail for failed requests, and no way to distinguish transient failures from data errors from SQL Server outages in production.

Authorization

There is no authentication or authorization. All endpoints — including the watchlist-backed screening and the historical verification audit data — are publicly accessible to any HTTP caller with network access to the service.

Version & Compatibility Risks

org.json:json:20140107 is pinned to a 2014 snapshot build — this is over a decade old and significantly behind current releases (the project has published releases up to 2024). Older org.json versions have had known NVD-tracked vulnerabilities (e.g. stack-overflow on deeply nested input, CVE-2022-45688). The javax.servlet-api:4.0.1 coordinates belong to the javax. namespace; Tomcat 10+ migrated to the jakarta. namespace (Jakarta EE 9+), so a container upgrade would require moving to jakarta.servlet-api and updating all imports. Java 8 (sourceCompatibility = JavaVersion.VERSION_1_8) reached end-of-public-updates from Oracle in March 2022; while commercial LTS support exists, this creates a technical debt item for cloud-native deployments.

Notable Observations

  • Extremely lean dependency footprint — the project has only 3 declared dependencies, which minimises transitive risk but also means all JSON processing, connection pooling, and request validation are handled manually or via JDBC directly.
  • No connection pooling library — KycConnectionFactory calls DriverManager.getConnection() on every request with no pool (e.g. HikariCP, DBCP2). Under load this will cause performance and resource exhaustion issues, and is a migration concern when targeting Azure-managed databases.
  • org.json version is critically outdated — version 20140107 (January 2014) should be upgraded to a current release (20240303 or later) to address known security vulnerabilities and benefit from a decade of bug fixes.
  • No logging framework — the application has no declared logging dependency (no SLF4J, Log4j2, or Logback). Errors are silently swallowed via empty catch (SQLException ignored) blocks, making observability and cloud-readiness a concern.

Data Architecture & Persistence Layer

ZavaKYCService persists data in a single Microsoft SQL Server database (ZavaBankDB) via raw JDBC with no ORM framework; the schema consists of two application-managed tables.

> Note: There is no formal foreign-key constraint between KYCWatchlist and KYCVerification. The relationship above is logical — the verification status (PASS/REVIEW) is derived at runtime by querying KYCWatchlist and the result is stored in KYCVerification; no referential integrity is enforced at the database level.

No batch queries, named queries, stored procedures, or transaction management annotations are used. Each request is effectively auto-committed via the default JDBC connection behaviour.

Caching Strategy

No caching layer is implemented. Every API request results in a direct JDBC round-trip to SQL Server. There is no in-process cache (Caffeine, EhCache), distributed cache (Redis, Memcached), or query-level second-level cache. Watchlist data, which is relatively static, is fetched from the database on every call to /api/kyc/verify.

Data Ownership Boundaries

The service uses a single shared ZavaBankDB database instance. There is no database-per-service or logical schema separation. The two tables (KYCWatchlist, KYCVerification) are exclusively owned and accessed by ZavaKYCService. There are no cross-service data access patterns — the service does not query or share data with any other application.

Write path: POST /api/kyc/verify → inserts into KYCVerification (and optionally reads KYCWatchlist).

Read path: GET /api/kyc/status/{customerId} → reads latest row from KYCVerification.

No CQRS, event sourcing, outbox pattern, or read-replica routing is configured.
