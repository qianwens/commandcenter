Data Storage & External Services

The application does not use a relational database or cache. Persistence is achieved through the local file system: incoming files land in /shared/filedrop, successfully processed files are moved to /shared/filedrop/processed, and files that fail processing are moved to /shared/filedrop/error. The sole external service dependency is a RabbitMQ broker (AMQP on port 5672), which receives published records on the filedrop.events topic exchange and failed-file notifications on the zava.dlx fanout dead-letter exchange. All connection parameters (host, port, virtual host, credentials, exchange names) are supplied via app.config or environment variable overrides.

Key Architectural Decisions

  • Poll-and-Watch hybrid: Combines a FileSystemWatcher for immediate event-driven detection with a periodic polling loop to catch files missed by OS events, making the bridge resilient to file system notification gaps.
  • Batch-per-poll cycle: Files queued during a polling interval are collected into a single batch and processed under one RabbitMQ connection and channel, reducing connection overhead.
  • Configuration hierarchy: Environment variables take precedence over app.config <appSettings> keys, enabling twelve-factor-style deployment without modifying the config file.

DTOs & Contracts

There are no formal DTO classes. The outbound message contract is constructed dynamically at runtime inside BuildPayload and serialized with JavaScriptSerializer. The effective JSON shape published to filedrop.events is:


{
  "sourceFile": "<original file name>",
  "processedAt": "<yyyy-MM-ddTHH:mm:ss>",
  "record": {
    "<header or field name>": "<value>",
    "...": "..."
  }
}

For CSV files, the record object keys are the column headers from the first row of the file. For fixed-width files (.txt, .dat), the record object always contains the following five keys: RecordType, CustomerId, AccountId, Amount, Details.

The dead-letter message published to zava.dlx has this shape:


{
  "sourceFile": "<original file name>",
  "error": "<exception message>",
  "failedAt": "<yyyy-MM-ddTHH:mm:ss>"
}

No OpenAPI/Swagger specification, protobuf .proto schema, or GraphQL schema exists. Serialization uses the legacy System.Web.Extensions.JavaScriptSerializer; this class is not available in .NET 5+ and must be replaced during migration.

Communication Patterns

Asynchronous — AMQP via RabbitMQ:

  • ZavaQueueBridge publishes messages to two exchanges declared on the broker:
    • filedrop.events — type topic, durable. Routing key is derived from the file extension: filedrop.csv, filedrop.txt, filedrop.dat, or filedrop.unknown. Downstream consumers can bind queues using wildcard patterns (filedrop.# or filedrop.csv).
    • zava.dlx — type fanout, durable. Receives error envelopes when file processing fails. Any queue bound to this exchange receives all dead-letter notifications.
  • Connection parameters (host, port, vhost, user, password) are read from app.config or environment variable overrides. AutomaticRecoveryEnabled is explicitly set to false, meaning transient broker disconnections during a poll cycle will surface as exceptions and cause the affected batch to be retried on the next poll interval.

File-System Input (pseudo-async):

  • A FileSystemWatcher monitors the drop directory for Created, Changed, and Renamed events, feeding paths into a shared in-memory HashSet<string> (guarded by a lock). A foreground poll loop drains this set on each tick, additionally scanning the directory with Directory.GetFiles to catch any files the watcher may have missed. Poll interval defaults to 5 000 ms and is configurable.

Resilience:

  • No circuit-breaker or Polly policy is implemented.
  • File-lock retry: WaitForFileReady attempts to open the file up to 5 times with 250 ms back-off before processing, guarding against partially-written files.
  • Per-file error isolation: a try/catch around ProcessSingleFile ensures one failing file does not abort the rest of the batch; the failed file is dead-lettered and moved to the error directory.
  • No retry on RabbitMQ publish failures — a publish exception propagates up to the per-file catch block and the file is dead-lettered.

Service Discovery:

  • None. The RabbitMQ host is resolved by static hostname / IP from configuration (RABBITMQ_HOST env var or RabbitMQ.Host app setting). No Consul, Eureka, or Kubernetes DNS-based discovery is used.

Security Posture:

  • No TLS/SSL is configured on the AMQP connection; traffic between ZavaQueueBridge and the RabbitMQ broker is unencrypted.
  • Authentication relies on username/password credentials stored in app.config (RabbitMQ.User / RabbitMQ.Password) with support for environment variable injection. There is no secrets management integration (e.g., Azure Key Vault, HashiCorp Vault).
  • No inbound authentication or authorization is applicable as the service has no HTTP surface.

Startup Dependency Chain

ZavaQueueBridge has an implicit runtime dependency on the RabbitMQ broker but no automated startup-wait mechanism:

  1. RabbitMQ Broker — must be reachable on RABBITMQ_HOST:RABBITMQ_PORT before the first poll cycle attempts to publish. There is no dockerize, Docker Compose depends_on with health check, or readiness probe.
  2. ZavaQueueBridge — starts immediately; if RabbitMQ is unavailable when the first non-empty poll cycle runs, a connection exception is thrown. The exception is caught at the batch level, logged to stdout, and the process continues. The next poll cycle will retry the connection.
  3. Shared file-drop directory — EnsureDirectories() creates the drop, processed, and error directories at startup if they do not exist. If the parent mount is not available, Directory.CreateDirectory will throw and the process will crash.

No Kubernetes readiness/liveness probes, Spring Cloud Config retry, or Consul health-check integration is configured.

Core Business Workflows

ZavaQueueBridge provides a file-drop-to-message-queue bridge for ZavaBank: it detects incoming transaction and data files placed in a shared directory, parses their records, and publishes each record as an event to a message broker so that downstream banking systems can consume them asynchronously.

Primary Workflows

Workflow 1: File Detection

Trigger: A file is created, modified, or renamed in the configured drop directory.

Steps:

  1. FileSystemWatcher raises a Created, Changed, or Renamed OS event.
  2. The event handler calls QueuePendingFile with the full file path.
  3. IsCandidateFile is evaluated: the path must have a .csv, .txt, or .dat extension and must not be named processed or error (subdirectory guard).
  4. If the file is a valid candidate it is added to the thread-safe PendingFiles HashSet. Duplicate paths are automatically deduplicated.
  5. On the next poll tick, PollFiles also scans the directory with Directory.GetFiles to pick up any files the watcher may have missed (e.g., files present before the watcher started, or missed due to OS buffer overflow).

Business Rule: Only .csv, .txt, and .dat files are eligible for processing. All other extensions are silently ignored.

Workflow 2: File Processing & Event Publishing

Trigger: A non-empty batch of pending file paths is drained from PendingFiles at the start of a poll cycle.

Steps:

  1. A single RabbitMQ connection and channel are opened for the entire batch.
  2. Both exchanges (filedrop.events topic, zava.dlx fanout) are declared idempotently on the channel.
  3. For each file in the batch:

a. File readiness check — WaitForFileReady attempts to open the file exclusively up to 5 times (250 ms apart). This guards against partially-written files.

b. Parsing — ParseRecords dispatches to ParseCsv (for .csv) or ParseFixedWidth (for .txt/.dat).

c. Payload construction — for each parsed record, BuildPayload wraps it with sourceFile and processedAt metadata and serialises to JSON.

d. Routing key derivation — BuildRoutingKey produces filedrop.<extension> (e.g., filedrop.csv, filedrop.txt).

e. Publish — each JSON message is published to filedrop.events with Persistent=true and ContentType=application/json.

f. Archival — the source file is moved to the processed/ subdirectory. If a name collision exists, a timestamp prefix is added.

  1. The RabbitMQ connection is disposed after the batch completes.

Business Rule: Every record in a file is published individually; one file can produce many events. The file is only moved to processed/ after all its records have been published without error.

Workflow 3: Error Handling & Dead-Lettering

Trigger: Any exception thrown during file processing (file unreadable after retries, parse failure, publish failure, etc.).

Steps:

  1. The exception is caught at the per-file level inside PollFiles.
  2. A dead-letter JSON message containing sourceFile, error (exception message), and failedAt timestamp is published to the zava.dlx fanout exchange.
  3. The source file is moved to the error/ subdirectory (with timestamp prefix on collision).
  4. Processing continues with the next file in the batch — one failing file does not abort the others.
  5. The error is logged to stdout.

Business Rule: A file that fails for any reason is quarantined in error/ and a notification is sent to zava.dlx. The file is not retried automatically; manual intervention is required to reprocess it.

Workflow 4: Graceful Shutdown

Trigger: Ctrl+C signal (SIGINT) or SIGTERM (process exit).

Steps:

  1. OnCancelKeyPress sets _keepRunning = false and cancels the signal (preventing immediate termination).
  2. The main poll loop exits after the current poll cycle completes (it checks _keepRunning between files within a batch as well).
  3. The FileSystemWatcher is disabled and disposed.
  4. The process exits cleanly.

Business Rule: Any batch currently in-flight at shutdown time is completed before the process exits. No partial batches are abandoned.

Cross-Service Data Flows

ZavaQueueBridge is a single-service application and does not orchestrate or aggregate data from multiple backend services. All cross-system data flow is one-directional:

  • Upstream producers (not part of this repository) write files to the shared drop directory (/shared/filedrop). The contract with producers is the file format: CSV with a header row, or an 80-character fixed-width layout with fields at defined offsets.
  • ZavaQueueBridge reads those files, transforms each record into a JSON event, and publishes to filedrop.events.
  • Downstream consumers (not part of this repository) bind their own queues to filedrop.events and process events independently. The routing key (filedrop.csv, filedrop.txt, filedrop.dat) allows consumers to subscribe selectively by file type.
  • Error consumers may bind queues to zava.dlx to receive dead-letter notifications and trigger alerting or manual remediation workflows.

There is no circuit breaker for the RabbitMQ connection. If the broker is unavailable during a publish, the exception propagates to the per-file catch block, the file is dead-lettered (which will also fail silently if the broker is still down), and the file is moved to error/. The next poll cycle will attempt a fresh connection.

State Transitions

Each FiledropFile progresses through the following lifecycle:


Detected → [Pending] → Processing → Published → Processed (moved to /processed)
                                  ↘ Failed → Dead-lettered → Error (moved to /error)
  • A file can only be in one state at a time (the HashSet deduplication prevents double-processing within the same poll cycle).
  • There is no retry from the Error state — manual reintroduction of the file into the drop directory is required.

Error Handling & Compensating Actions

  • Per-file isolation: Exceptions are caught at the individual file level; a single file failure never aborts processing of sibling files in the same batch.
  • Dead-letter event: Every failure results in a zava.dlx message containing the file name and exception message, enabling downstream alerting or monitoring queues.
  • File quarantine: Failed files are moved to error/ to prevent repeated detection by Directory.GetFiles on the next poll cycle.
  • No automatic retry: The application does not re-enqueue failed files. Retry logic must be implemented at the operational level (e.g., moving files back to the drop directory after investigating the cause).

Cross-Cutting Concerns

  • Transactions: There is no distributed transaction. Publish and file-move are independent operations. If the publish succeeds but File.Move fails, the message will have been sent to RabbitMQ without the file being archived — a potential duplicate-event risk on process restart.
  • Thread safety: PendingFiles (HashSet<string>) is guarded by a lock(PendingLock) statement. The FileSystemWatcher callback and the poll loop run on different threads.
  • Shutdown safety: The _keepRunning flag is declared volatile to ensure the poll loop and batch-processing inner loop observe the updated value across threads without a memory barrier.
  • Audit / logging: All significant business events (startup, file count per cycle, publish count per file, file moves, errors) are written to stdout with a [QueueBridge] prefix. No structured logging or correlation IDs are present.
  • Authorization: No authorization model is implemented. Any process with write access to the drop directory can inject files, and any AMQP client with the correct credentials can connect and consume from the exchanges.

Version & Compatibility Risks

The application targets .NET Framework 4.8, which is in long-term maintenance mode and is Windows-only in its native form (the Docker image works around this with Mono 6.12, itself in maintenance). The single NuGet dependency, RabbitMQ.Client 5.2.0, was released in 2019 and is several major versions behind the current stable line (7.x); the newer client API introduces breaking changes (e.g., async-first model, removal of IModel in favour of IChannel). The use of System.Web.Extensions.JavaScriptSerializer is a significant risk: this class is not present in .NET 5+ and has no direct equivalent — migration requires replacing it with System.Text.Json or Newtonsoft.Json. System.Configuration.ConfigurationManager is available as a NuGet package on modern .NET but requires an explicit dependency addition. Overall, every non-messaging dependency relies on framework APIs that either do not exist or require a NuGet shim on .NET 6/8.

Notable Observations

  • Single external NuGet package: The extremely lean dependency tree reduces upgrade complexity in terms of third-party libraries, but the real migration burden is in framework-built-in APIs (JavaScriptSerializer, ConfigurationManager, FileSystemWatcher configuration model) that behave differently or require shims on modern .NET.
  • RabbitMQ.Client 5.2.0 is outdated: The current stable release is 7.x (with a 6.x LTS stream); upgrading involves API-breaking changes to how connections, channels, and publishing work — a rewrite of PollFiles, ProcessSingleFile, and PublishDeadLetter is required.
  • Mono 6.12 as runtime container: Shipping via Mono is a workaround for running .NET Framework on Linux; Mono 6.12 is end-of-life. Migrating to .NET 8 + official mcr.microsoft.com/dotnet/runtime base image would eliminate this risk and provide a supported, maintained container base.
  • No logging framework: The application uses raw Console.WriteLine for all output. This limits structured logging, log-level filtering, and integration with cloud-native observability stacks. Adding Microsoft.Extensions.Logging or Serilog is recommended during migration.

Caching Strategy

No caching layer is implemented. The only in-process state is a HashSet<string> (PendingFiles) protected by a lock statement that holds file paths queued by the FileSystemWatcher between poll cycles. This is a transient coordination structure, not a cache — it is drained and cleared on every poll tick and does not survive process restarts.

Data Ownership Boundaries

ZavaQueueBridge is a single-service application with no peer services sharing its data stores. Data flows strictly one way:

  1. Read — files are read from a shared volume mount (/shared/filedrop); other services (e.g., upstream producers) write files into this directory.
  2. Transform — file contents are parsed into record dictionaries in-process with no persistent intermediate store.
  3. Write — serialized JSON messages are published to RabbitMQ exchanges; downstream consumers own the durable queues bound to those exchanges and are responsible for their own persistence.

There is no shared relational database, no cross-service ORM access, and no CQRS pattern. The service has no read model separate from its write path.

The application processes files whose content is determined entirely by the upstream producer. The fixed-width record schema contains fields that are potentially sensitive in a banking context:
